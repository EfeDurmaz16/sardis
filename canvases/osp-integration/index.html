OSP Integration

Open Service Protocol (OSP) is an open standard — separate from Sardis — for AI agents to discover, provision, rotate, and deprovision developer services. Sardis is a reference payment rail inside OSP, declared as sardis_wallet in any provider's accepted_payment_methods. This canvas shows the bridge: what OSP is, why Sardis is one of its rails, and the code that wires them.

TL;DR OSP is to service provisioning what MCP is to tool access. A provider publishes /.well-known/osp.json; an agent calls POST /osp/v1/provision; credentials come back Ed25519-encrypted. OSP is payment-rail agnostic — Sardis plugs in via @sardis/osp-integration, providing the wallet, escrow, ledger, and an MCP server that exposes nine OSP tools to Claude / Cursor / other MCP clients.
Motivation

Why OSP exists.

Every managed service today has its own proprietary onboarding flow. Agents can't fill signup forms, can't solve CAPTCHAs, can't verify emails. The alternatives are invite-only ecosystems (Stripe Projects, March 2026) or custom-per-provider glue. OSP is the open-standard third path.

Problem 1
Manual signup is the wrong primitive
Browsers, CAPTCHAs, email verification — interaction patterns agents cannot automate without brittle scraping. Every new service is bespoke integration work.
Problem 2
Platform lock-in beats open coordination
Stripe Projects is invite-only, Stripe-rail-only. OSP is Apache 2.0; any provider can implement it, any agent can consume it, any rail can settle it.
Problem 3
Reusable credentials don't fit agent threat models
OSP's core thesis: "Reusable credentials are the wrong primitive. Every service interaction should be a first-class protocol operation — discoverable, provisionable, rotatable, auditable."
Problem 4
No standard place to settle a bill
Agents can reason about a service spec but have no universal way to pay for it. OSP declares accepted_payment_methods; Sardis is one of them alongside x402, MPP, Stripe SPT, free.
Protocol Flow

Discover → provision → rotate → deprovision.

Four operations define the OSP lifecycle. Every provider exposes them under the same URL scheme; every agent speaks the same JSON. Spec lives at ~/osp/spec/osp-v1.0.md (v1.1 draft). 

sequenceDiagram
    participant A as Agent
    participant P as Provider (e.g. Supabase)
    participant S as Sardis (payment rail)

    A->>P: GET /.well-known/osp.json
    P-->>A: manifest { offerings, tiers, accepted_payment_methods }
    A->>A: Pick offering + tier + payment_method="sardis_wallet"

    Note over A,S: Escrow leg (optional but recommended)
    A->>S: EscrowManager.hold(amount, release_condition)
    S-->>A: escrow_id + signed proof

    A->>P: POST /osp/v1/provision { offering_id, tier_id, agent_public_key, payment_proof: escrow_id }
    P->>S: verify escrow_id
    S-->>P: held=true, amount ok
    P->>P: allocate resource_id
    P-->>A: { resource_id, credentials (Ed25519-encrypted) }

    Note over A,S: Settlement leg
    P->>S: release(escrow_id) on successful provision
    S->>S: ledger append (credit provider, debit agent mandate)

    opt Rotation
        A->>P: POST /osp/v1/rotate/{resource_id}
        P-->>A: new encrypted credentials
    end

    opt Deprovision
        A->>P: POST /osp/v1/deprovision/{resource_id}
        P-->>A: receipt
    end
The Bridge

Where Sardis fits in an OSP flow.

OSP's provision-request.schema.json enumerates seven payment methods: free, sardis_wallet, stripe_spt, x402, mpp, invoice, external. Sardis is the first-class crypto-native rail. 

// Provider manifest — /.well-known/osp.json
{
  "provider_id": "supabase.com",
  "offerings": [{
    "offering_id": "supabase/postgres",
    "tiers": [
      { "tier_id": "free", "price": { "amount": "0", "currency": "USD" } },
      { "tier_id": "pro",  "price": { "amount": "25", "currency": "USD", "interval": "P1M" } }
    ]
  }],
  "accepted_payment_methods": ["free", "sardis_wallet", "stripe_spt", "x402"]
}

When the agent sends payment_method: "sardis_wallet", the payment_proof is a Sardis escrow ID or charge intent. The provider verifies it out-of-band with the Sardis API (/api/v2/escrow/{id}) before allocating the resource. On successful provisioning the escrow releases; on failure the hold is refunded.

The Package

@sardis/osp-integration — three modules.

Lives at ~/osp/sardis-integration/ (peer to the OSP core, shipped as its own npm package). Three modules, each a clean seam.

Module · payment/
Wallet · escrow · ledger · charge
SardisWalletClient, EscrowManager, Ledger, and the typed shapes for SpendingMandate, EscrowHold, ChargeIntent, LedgerEntry. The minimum surface a provider needs to accept sardis_wallet.
Module · mcp/
Nine OSP tools for MCP clients
sardis-mcp-server.ts exposes: discover_services, provision_service, deprovision, rotate_credentials, check_status, get_usage, estimate_cost, list_skills, env_generate. Makes Claude/Cursor first-class OSP consumers.
Module · cli/
Sardis CLI extension
commands.ts + bridge.ts. Adds sardis osp discover, sardis osp provision, etc. — same bridge as the MCP server, command-line shape. The payment layer is shared.
Shared Cryptography

Ed25519 everywhere.

Both OSP and Sardis settled on Ed25519 for identity + envelope encryption. This is not coincidence — it means a single agent keypair can simultaneously sign a Sardis mandate, prove identity to an OSP provider, and decrypt the returned credential bundle.

Strategic Framing

Two open standards, one author, one platform thesis.

Sardis and OSP are deliberately separate projects. Sardis is a product (fintech rail, VC-fundable). OSP is a standard (Apache 2.0, zero revenue target). They reinforce each other because every OSP provider that accepts sardis_wallet puts Sardis in front of another agent cohort — and every Sardis agent that wants to provision a DB, a logger, a feature flag service gets a clean OSP path to do so.

Sardis — the product layer
Non-custodial wallet, MPC signing, spending mandates, KYC/AML, on-chain anchor, 15+ framework SDKs. Revenue = platform fee + enterprise compliance tier. Live on Base + Tempo mainnet.
OSP — the standards layer
Protocol spec (9,600+ lines, v1.1 draft). Apache 2.0. SDKs in Rust, TypeScript, Python, Go. Rebranded public umbrella: Pact. Maximum adoption, zero revenue ambition — the moat is in being the default spec that Sardis (and others) implement.
