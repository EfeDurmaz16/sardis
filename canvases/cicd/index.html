CI / CD

Twenty-five GitHub Actions workflows guard the repo. Four classes: tests on every PR, security scans on every push and weekly, deploys on merge to main, publishes on tag. One branch is production (main). Everything else is preview.

TL;DR main is the only long-lived branch. Every PR runs the full CI matrix (Python, contracts, TS SDK, and landing) plus security. Merge to main triggers path-scoped deploys: API → Cloud Run us-east1, Landing → Vercel. Releases to PyPI and npm fire on version tags. Uptime pings every 10 minutes.
Overview

Branch strategy and gating.

The default branch is main. PRs against it run the CI and security suites but do not publish. Merges to main trigger the deploy workflows that match changed paths — only the pieces that actually changed redeploy.

What runs on a PR
ci.yml (Python lint+test, contracts strict + gas, TS SDK, landing build), test-api.yml (sardis-api scoped tests), codeql.yml (Python + JS/TS), secret-scan.yml (gitleaks), security-scan.yml (pip-audit + Trivy + Bandit), fuzz.yml (atheris against spending_policy). Path filters skip irrelevant workflows.
What runs on merge to main
deploy-api-cloudrun.yml, deploy-landing.yml, deploy.yml — each gated by a paths: filter so a docs-only merge doesn't redeploy the API. Verification step (curl /health) runs after each Cloud Run deploy.
What runs on a version tag
release-python-sdk.yml (PyPI / TestPyPI — py-sdk-v*), release-python-integrations.yml (batch publish for langchain/crewai/adk/agent-sdk/connect), release-npm.yml (mcp-v*, sdk-js-v*, ai-sdk-v*, connect-js-v*), publish.yml, sign-artifacts.yml. Each verifies pyproject/package.json version matches the tag, runs tests, builds, publishes, then smoke-installs from the registry.
What runs on a schedule
monitoring.yml every 10 min (synthetic /health probe + Slack notify on failure); uptime*.yml (uptime, uptime-response-time, uptime-graphs, uptime-static-site, uptime-summary); nightly-sandbox-e2e.yml 04:20 UTC; codeql.yml weekly Monday; scorecard.yml weekly Monday; fuzz.yml weekly Saturday.

Gotchas

Deploy lessons paid for in incidents.

Things that silently failed at least once and now have permanent fixes in the deploy workflow and container config. Worth knowing before you touch Cloud Run config.

Gunicorn 25.x — use call syntax, not --factory
Launch the API as "server.main:create_app()". The --factory flag was removed in gunicorn 25.x; using it produces a cryptic boot failure that looks like a module-not-found error. Docker CMD, cloudbuild.yaml, and deploy-api-cloudrun.yml encode the call form.
Install gunicorn + uvicorn inside .venv
The container's system Python shouldn't host gunicorn — uv installs into .venv, so gunicorn must live there too. Otherwise Cloud Run boots with the wrong interpreter and the app never binds its port, tripping the startup probe.
Always gcloud ... --update-env-vars, never --set-env-vars
--set-env-vars silently wipes every other env var on the service. Use --update-env-vars to merge when changing a live service by hand; private deployment helper scripts live outside the public repo.
Workload Identity Federation, not SA keys
deploy-api-cloudrun.yml authenticates through google-github-actions/auth with GCP_WORKLOAD_IDENTITY_PROVIDER + GCP_SERVICE_ACCOUNT. No JSON key lives in GitHub secrets. If the WIF provider secret is missing, the job fails loudly (older behaviour silently skipped — backend changes never shipped for weeks).
