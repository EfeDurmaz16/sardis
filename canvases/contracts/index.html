Smart Contracts

Sardis is deliberately thin on-chain. The heavy lifting — policy evaluation, mandate verification, KYC, compliance — happens off-chain where it can be iterated safely. On-chain we only deploy what needs public, auditable, tamper-evident behaviour: identity registries, an evidence anchor, an escrow-based job primitive (ERC-8183), and a refund protocol.

TL;DR Nine .sol files live under contracts/src/. Two are deprecated on purpose (SardisPolicyModule → Zodiac Roles; SardisVerifyingPaymaster → Circle Paymaster) — kept for reference, not deployed. The rest implement ERC-8004 (agent identity), ERC-8183 (job escrow), a minimal ledger anchor, and Circle's audited RefundProtocol vendored in-tree.

Principles

Why so little on-chain?

Prefer audited standards over bespoke contracts
Wallet custody: Safe v1.4.1 (pre-deployed on every EVM chain, $100B+ TVL). Policy module: Zodiac Roles (0x9646fDAD…). Paymaster: Circle Paymaster (0x0578cFB2…). Sardis ships no custom money-handling contract if an audited one fits — zero audit surface at MVP.
On-chain for evidence, off-chain for decisions
Policy decisions live in Postgres (migration 041) because policies change. What goes on-chain is the Merkle root of a batch of those decisions (SardisLedgerAnchor) — tamper-evidence without tying policy logic to chain latency or upgrade ceremony.
Standards-compliant identity & reputation
Agent identity is ERC-8004 (ERC-721 token + reputation + validation registries). A wallet can prove "I am agent #42 in this registry" without trusting a Sardis API. Reputation is submitted by clients, signed on-chain, and queryable by any counterparty.
ERC-8183 for commerce, not custom escrows
Escrow-based agent commerce follows the ERC-8183 spec with a strict state machine and selector-based hooks (IACPHook). Easier for counterparties to integrate than a Sardis-only escrow; easier to audit.
Hook-dispatch over inheritance
SardisJobManager routes hook calls through IACPHook with a gas limit — before-hooks can block, after-hooks are fire-and-forget. Keeps hook failures from griefing the core flow while still letting composability extensions run.
Apache 2.0 for RefundProtocol
RefundProtocol.sol is Circle's audited implementation, vendored in-tree under Apache 2.0 — not a rewrite. The point is to reuse the audit, not re-audit.

SardisLedgerAnchor.sol
contracts/src/SardisLedgerAnchor.sol · Solidity ^0.8.20
MIT Production Ownable
Stores Merkle roots of audit-log batches for tamper evidence. Minimal by design — the ledger itself stays off-chain in Postgres; only the root crosses the chain boundary. anchor(root, anchorId) writes once, never twice; verify(root) returns the timestamp (0 if never anchored). This is the contract that makes "trust our logs" obsolete.
Key API
anchor(bytes32 root, string anchorId) · verify(bytes32) → uint256 timestamp
Event
Anchored(bytes32 indexed root, string anchorId, uint256 timestamp)
Upstream
sardis-ledger batches → on-chain call from ops service.
SardisIdentityRegistry.sol
Solidity ^0.8.24 · implements ERC-8004 Identity
MIT ERC-8004 ERC-721 EIP-712 ERC-1271
ERC-8004 Identity Registry — on-chain agent identity as an ERC-721 token. Global addressing: eip155:{chainId}:{registryAddress}#{agentId}. Three register() overloads (URI + metadata, URI only, bare). Agent-wallet linking verified via EIP-712 signature (EOA) or ERC-1271 (smart contract). Wallet is cleared on token transfer so the new owner has to re-verify.
Key API
register(...) · setAgentWallet(agentId, wallet, deadline, sig) · setMetadata(agentId, key, value)
Invariant
Reserved key agentWallet cannot be written via setMetadata — only via signed flow.
Spec
eips.ethereum.org/EIPS/eip-8004
SardisReputationRegistry.sol
Solidity ^0.8.24 · implements ERC-8004 Reputation
MIT ERC-8004
Client-submitted on-chain feedback for agents. Each feedback is int128 value + decimals + tags + endpoint + off-chain URI + hash. Revocation is soft (marked, never deleted). Agent owners (or approved operators) can submit response records pointing at their side of the story.
Rules
Submitter MUST NOT be the agent owner; valueDecimals in [0, 18]; only original submitter can revoke.
Linked to
SardisIdentityRegistry (ERC-721) — feedback keyed by agentId.
SardisValidationRegistry.sol
Solidity ^0.8.24 · implements ERC-8004 Validation
MIT ERC-8004
Agent owners request validation from a designated validator; the validator responds with a score (0–100). Records are queryable by requestHash. Validators can respond multiple times to update the score.
Record shape
validatorAddress · agentId · response (uint8) · responseHash · tag · lastUpdate.
Immutable
identityRegistry address is set at construction.
SardisJobManager.sol
Solidity ^0.8.24 · implements ERC-8183
MIT ERC-8183 Escrow ReentrancyGuard Pausable
Escrow-based job primitive for agentic commerce. Full ERC-8183 state machine: Open → Funded → Submitted → Completed/Rejected/Expired with claimRefund on expiry. Hook dispatch through IACPHook — a before hook called with HOOK_GAS_LIMIT whose revert blocks the action, and a fire-and-forget after hook whose revert is ignored. Bounded gas prevents hook-griefing.
State machine
Open → Funded (fund) · Open → Rejected (reject) · Funded → Submitted (submit) · Submitted → Completed (complete) · Submitted → Rejected (reject) · Funded/Submitted → Expired (claimRefund).
Caps (MVP)
1% fee cap, USDC-only, trust hooks disabled post-launch — relaxable post-audit.
Spec
eips.ethereum.org/EIPS/eip-8183
SardisJobRegistry.sol
Solidity ^0.8.24
MIT Reputation Ownable
On-chain reputation store for ERC-8183 agents. Tracks completed / rejected jobs, earnings, spending per agent. Only authorized writers (the Job Manager, trusted hooks) can update records. Trust score is computed on-chain in basis points (0–10000): baseScore = completed / (completed + rejected) × 10000 + up to a 10% volume bonus capped at 10000.
Auth model
authorizedWriters mapping — only owner can toggle.
Companion of
SardisJobManager — Manager calls into Registry on every complete/reject.
RefundProtocol.sol
Solidity ^0.8.24 · Circle, vendored
Apache 2.0 Circle (audited) EIP-712
Circle's RefundProtocol, vendored in-tree. Time-locked payments (up to 180 days) with arbiter-enabled early withdrawal authorised by EIP-712 signed EarlyWithdrawalByArbiter messages. Sardis uses it for dispute-resolved refunds so neither buyer nor merchant has to trust Sardis custody.
Payment struct
to · amount · releaseTimestamp · refundTo · withdrawnAmount · refunded.
Max lockup
180 days (MAX_LOCKUP_SECONDS).
Why in-tree
Pin version under Sardis build/audit; swap upstream fixes with explicit review.
SardisPolicyModule.sol
Solidity ^0.8.20 · Safe module (reference only)
Deprecated MIT Safe module
Safe module that enforces spending policies for AI agent wallets — per-tx + daily limits, co-sign mode (10× limit when co-signed by Sardis), merchant allow/deny lists, token allowlist, pause/unpause. Superseded by the pre-deployed, audited Zodiac Roles module at 0x9646fDAD06d3e24444381f44362a3B0eB343D337. Kept for reference; not deployed to mainnet for MVP (no custom audit dependency).
Status
Do-not-deploy. Any agent-wallet policy enforcement routes through Zodiac Roles.
SardisVerifyingPaymaster.sol
Solidity ^0.8.20 · ERC-4337 paymaster (reference only)
Deprecated MIT ERC-4337
ERC-4337 verifying paymaster with wallet allowlist, per-op cap, and daily sponsor cap. Validation optionally consumes off-chain verifier signatures in paymasterData. Superseded by Circle Paymaster (0x0578cFB241215b77442a541325d6A4E6dFE700Ec, permissionless, USDC gas). Kept for reference; not deployed.
Today
Primary: Circle Paymaster. Backup: Pimlico (sponsor model).
