Settlement & Chains

Once a transaction is policy-approved and signed, it has to actually move money — and money moves on many different rails. Sardis supports six EVM chains, Tempo, Circle's Arc, and fiat off-ramps through Stripe, Striga, and Bridge. This canvas walks the rail map, the cross-chain path via CCTP, and the reconciliation machinery that keeps it all honest.

TL;DR Chains are defined in ChainConfig (packages/sardis-core/src/sardis_v2_core/config.py); token contracts live in tokens.py. ChainExecutor in packages/sardis-chain/src/sardis_chain/executor.py owns submission; CCTP v2 handles cross-chain USDC; Stripe / Striga / Bridge handle fiat off-ramp. Settlement state is explicit (migration 048), provider-rail is tracked (053), and a persistent reconciliation queue (057) catches anything that falls off.

Base

The primary production rail.

Why Base is default
Coinbase-native L2, Circle Paymaster v0.8 works permissionlessly with no API key, sub-cent fees, mature bundler ecosystem (Pimlico + Alchemy), Safe v1.4.1 singleton pre-deployed. For USDC-denominated agent flows, Base is the path of least resistance.
RPC strategy
Alchemy primary via SARDIS_BASE_RPC_URL, public mainnet.base.org as fallback. ChainRPCClient in executor.py handles retries and failover between endpoints.
Checkout default
SARDIS_CHECKOUT_CHAIN defaults to base. "Pay with Sardis" resolves to a Base USDC transfer unless the merchant explicitly opts into another rail.
Smart contracts here
SardisLedgerAnchor.sol, ERC-8183 stack, reference Safe modules — all intended first for Base mainnet (and mirrored to Tempo). See canvas 06 · Contracts.

Cross-chain

CCTP v2 — USDC across chains without bridges.

Circle Cross-Chain Transfer Protocol v2: burn USDC on source, mint USDC on destination. No wrapped assets, no pooled liquidity, no bridge counterparty risk. Sardis uses it wherever a payment starts on one chain and settles on another.

Implementation
packages/sardis-chain/src/sardis_chain/cctp.py — CCTPBridgeService, BridgeTransfer, BridgeStatus
Constants
cctp_constants.py — domain IDs, TokenMessenger / MessageTransmitter addresses per chain
Forwarding helper
cctp_forwarding.py — destination-chain forwarding so the minted USDC lands where the agent actually wants it, not at the messenger address
Unified addresses
CCTP v2's big win — same token messenger address on every chain, so integrations aren't n² chain pairs.
Typical use
Agent holds USDC on Base; merchant is on Polygon; CCTPBridgeService.bridge_usdc() burns on Base, mints on Polygon, forwards to merchant address in one flow.

    participant Ex as ChainExecutor
    participant TMb as TokenMessenger (Base)

Fiat off-ramps

USDC → merchant bank, three providers.

Merchants eventually want fiat in their bank account. Sardis doesn't pick one provider — it picks the cheapest path per currency and region. The selector lives in CrossCurrencySettlementService.

Stripe Connect
US / USD · ACH

USDC → USD → merchant bank. Connected Account model; Sardis handles the conversion and Stripe handles the ACH settlement.

Striga
EU / EUR · SEPA + vIBANs

EURC → EUR → SEPA. Per-merchant vIBANs so inbound SEPA lands against a known merchant identity. Fee ~0.3%.

Bridge
Multi-rail · ACH / SEPA / PIX / Wire

Provider abstraction over bank-settlement rails worldwide. Fee ~0.5% on USDC → USD. Used when Stripe / Striga don't cover the geography.

Path selection

CrossCurrencySettlementService.get_settlement_quotes() enumerates viable paths per request, scores by total fee + estimated time, and returns quotes. select_optimal_path() picks the best. The path identifiers are stable: usdc_bridge_usd, usdc_grid_fx_eur, eurc_striga_eur, direct_stablecoin. 

flowchart LR
    P[Payment settles in USDC] --> Q{Merchant currency?}
    Q -- USD (US) --> S1[Stripe Connect: USDC → USD → ACH]
    Q -- EUR (EU) --> S2[Striga: EURC → EUR → SEPA]
    Q -- EUR (fallback) --> S3[Bridge: USDC → Grid FX → EUR → SEPA]
    Q -- BRL --> S4[Bridge: USDC → PIX]
    Q -- USD (non-Stripe) --> S5[Bridge: USDC → ACH/Wire]
    Q -- Stablecoin --> S6[Direct: stablecoin-to-stablecoin, no FX]
FX

Quotes and cross-currency settlement.

When source and destination currency differ, a quote has to be recorded: the rate, the fee breakdown, the expected received amount. That quote is part of the payment's evidence record — disputes reference it directly.

Service
packages/sardis-core/src/sardis_v2_core/cross_currency_settlement.py — CrossCurrencySettlementService
Quote type
SettlementQuote: path, source & target currency/amount, exchange_rate, total_fee_cents, fee_breakdown, estimated_time_seconds, fee_percent property
Result type
SettlementResult: settlement_id, path, amounts, status, provider_tx_ids[], metadata
FX helpers
fx_quote.py (quote construction), fx_signer.py in sardis-chain (chain-side FX signing), Grid FX + Circle Cross-Currency adapters (circle_cross_currency.py).
Fee defaults (bps)
Bridge 50, Grid FX 50, Striga swap 30. Configurable at service init.
Tracking

Settlement is explicit, not best-effort.

"Payment succeeded" has two meanings: the chain accepted the tx, and the merchant's money actually arrived. The settlement_records table captures the full lifecycle of the second one.

settlement_records
Key fields
settlement_id (unique), intent_id, receipt_id, mode, status, amount, currency, fee, network_reference, credential_id
Card-path fields
authorization_status, capture_status, dispute_status, reversal_reference, liability_party
Timeline
initiated_at → confirmed_at → settled_at (or failed_at). expected_settlement_at powers the "stuck settlements" alert.
Indexes
idx_stl_intent, idx_stl_status(status, mode), partial idx_stl_pending (initiated/pending_confirmation only), partial idx_stl_disputes.
Provider rail

Which rail actually processed a payment.

The same logical payment can ride different provider rails depending on routing. Operators need to see which one in every row — for reconciliation, cost analysis, and incident response.

Migration
packages/api/migrations/053_provider_rail_tracking.sql
Columns on transactions
provider_rail VARCHAR(50), provider_payment_reference VARCHAR(255)
Valid values
stripe_spt, visa_tap, mastercard_agent_pay, circle_cpn, bridge_ach, bridge_sepa, bridge_pix
Index
idx_transactions_provider(provider_rail, created_at) — powers the "costs by rail this month" dashboard query.
Why it matters
When a provider has an incident, operators need a one-query answer to "which payments are currently on that rail?" — without this column, you're grepping JSON metadata at 3am.
Reconciliation

Persistent queue — because chains reorg and RPCs flake.

If the on-chain tx lands but the ledger append fails, we can't just forget about it. The reconciliation queue holds the half-written payment until the ledger append succeeds. Persistence means a process restart doesn't lose a cent.

Migration
packages/api/migrations/057_reconciliation_queue.sql
Table
reconciliation_queue
Schema
entry_type TEXT, payload_json JSONB, status (pending / processing / resolved / failed), retry_count, created_at, processed_at
Index
Partial idx_recon_queue_status WHERE status='pending' — the worker only ever scans pending rows.
Why it exists
Replaces a prior InMemoryReconciliationQueue that lost entries on restart. Any payment where ledger append fails after chain execution now persists its reconciliation state across process restarts.
Operator behavior
Alert fires when retry_count > N or status = 'failed'. Human-in-the-loop resolution.
