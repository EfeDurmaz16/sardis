Canvas 05 · Backend

Database Schema

One Neon Postgres database. 107 sequentially-numbered SQL migrations under apps/api/migrations/. No ORM is doing the heavy lifting — tables are defined by hand, in SQL, and applied in strict order. When Postgres is the source of truth for money, legibility beats abstraction.

TL;DR Migrations are the canonical record of how the schema evolved. Reading them in order is the fastest way to understand what Sardis does: identity first, then wallets, then policies, then ledger, then merchants, then mandates, then protocol-specific tables (Tempo, A2A, x402), and finally the ba_-prefixed better-auth surface.

Conventions

How this schema is written.

Hand-written SQL, applied in order
Migrations live in apps/api/migrations/ as NNN_description.sql. Applied with psql $DATABASE_URL -f <file>. A schema_migrations table guards against double-apply. No Alembic, no Django, no Drizzle — direct SQL keeps the diff honest.
Append-only, additive-only
Destructive changes (drop column, tighten constraint) must be proposed as a new migration with a rollback companion (see 001_initial_schema_rollback.sql). ADD COLUMN IF NOT EXISTS is used liberally so migrations are re-runnable on partially-applied dbs.
Three primary-key styles, chosen on purpose
uuid for records a human might quote (orgs, wallets, merchants). text prefixed IDs (mandate_…, appr_…, mst_…) for records an agent or operator will handle. bigserial only for high-churn internal queues (reconciliation_queue).
ba_ prefix for better-auth tables
The better-auth library manages its own schema (ba_user, ba_session, ba_apikey, …) and coexists with legacy Sardis auth tables. Prefix avoids collisions while the dual-path is live. See migrations 077, 097, 098.
Indexes travel with the columns that need them
Every migration that adds a frequently-queried column also adds the index. Migration 087 (missing_indexes) exists to clean up what slipped through. Query plans stay predictable.
JSONB for open-ended, columns for load-bearing
metadata, settings, merchant_scope are JSONB because shape evolves. Money columns (amount_per_tx, spent_total) are NUMERIC(20,6) because floats in finance cost real money.

Integrity

How the schema defends correctness.

Unique idempotency
merchant_checkout_sessions.idempotency_key has a partial UNIQUE index. Two concurrent creates with the same key collide at the database — there is no "last write wins" path that could double-pay a customer (migration 031).
Row locks at claim time
Checkout claim uses SELECT … FOR UPDATE NOWAIT — if another worker already holds the lock, the caller fails fast rather than waits, and the client retries. Prevents two settlement workers from racing the same session.
Enum types for state machines
approval_status, approval_urgency, CHECK constraints on kya_status, settlement_type. Invalid states become database errors, not runtime surprises.
Cascading deletes where ownership is clear
ba_session.user_id → ba_user(id) ON DELETE CASCADE; mandate_state_transitions.mandate_id → spending_mandates(id) ON DELETE CASCADE. Ownership deletions don't leave orphans.
Reconciliation queue survives restarts
If a ledger append fails after on-chain execution, the reconciliation entry lands in reconciliation_queue with status='pending' — not in RAM. A worker drains it on the next process boot (migration 057).
Policy evidence is cryptographically hashed
policy_versions.hash and policy_decisions.evidence_hash mean a policy decision can be proven-in-court without trusting the Sardis log pipeline. Migration 040 and 041.
