Canvas 07 · Flows

Payment Flow · end-to-end

One call — sardis.pay(to, amount, reason) — traverses fourteen distinct stages before it returns. This canvas walks every one, with the actual file behind it, the latency budget, and what happens when that stage fails closed.

TL;DR Every payment is four things in sequence: identity (who is calling), approval (is this allowed by policy + compliance + mandate), execution (get an MPC signature, broadcast), receipt (ledger row, Merkle batch, webhook). Fail-closed at every gate. Budget ≈ 2s p99 excluding chain confirmation.

Stage 1 · Agent surface

From Python line to HTTP request.

01
Agent calls sardis.pay() or a framework tool.
In LangChain the agent invokes a PayTool from sardis-langchain/toolkit.py. In the bare Python SDK the call is client.payments.create(to, amount, reason). In MCP it's a sardis_pay tool call from Claude Desktop / Cursor. All paths converge on the same REST call — no framework gets a privileged fast path.
file: packages/sardis-sdk-python/src/sardis_sdk/client.py file: packages/sardis-langchain/src/sardis_langchain/tools.py fail: raised on client; no network call yet
02
SDK marshals request, attaches API key.
Client reads SARDIS_API_KEY, serializes the intent to JSON, sets Authorization: Bearer sk_live_… and an idempotency key. TypeScript SDK does the same via @sardis/sdk with Zod validation.
auth: API key HMAC-hashed at rest (SHA-256) idempotency: client-generated uuid, enforced DB-side fail: network retry with same idempotency key — safe
Stage 2 · API gateway

FastAPI receives the request.

03
Middleware stack runs — CORS, CSP, HSTS, rate limit.
Request lands on Cloud Run, enters FastAPI composition root (sardis-api/main.py). Middleware applies canonical prod origins, CSP/HSTS, and the Redis-backed rate limiter. Rate limiter is fail-closed in prod — if Redis is unreachable the request is denied.
file: packages/api/src/sardis/main.py env: SARDIS_REDIS_URL required fail: 429 on limit; 503 if Redis down
04
require_principal extracts identity.
Dependency resolver hashes the incoming API key, looks up the principal (org + agent + scopes) from Postgres, attaches it to the request. Better-auth JWTs (EdDSA via JWKS) take the same slot for dashboard-originated calls. Missing/invalid key → 401, no downstream work.
file: packages/api/src/sardis/routers/pay.py principal: org_id, agent_id, scopes[], key_id fail: 401 — no orchestrator invocation
Stage 3 · The single gate

PaymentOrchestrator + PreExecutionPipeline.

Every path (wallets, mandates, ap2, pay) funnels through these two files. This is the "policy before execution" invariant at the code level.

05
PaymentOrchestrator.execute() enters.
The orchestrator receives the intent + principal. Builds a PipelineContext containing chain, token, mandate refs (if any), agent card, trust graph snapshot. Calls the pipeline.
file: packages/sardis-core/src/sardis_v2_core/orchestrator.py invariant: chain_executor is private — only orchestrator touches it
06
Spending policy check.
Evaluates natural-language-derived rules against the intent — merchant allowlist, daily cap, per-merchant cap, category block. Returns allow / deny / require_approval. Denial ends the flow here with a reason code that's written to policy_decisions.
file: packages/sardis-core/src/sardis_v2_core/spending_policy.py fail-closed: missing policy → deny
07
KYC / AML screen.
Hits sardis-compliance which calls Didit (KYC status cached) + Elliptic (AML/sanctions). Missing or expired KYC for the principal → deny. Sanctioned counterparty → deny + SAR queue.
file: packages/sardis-compliance/src/sardis_compliance/kyc.py file: sardis_compliance/pep.py, sar.py fail-closed: compliance provider down → deny
08
Mandate delegation check.
If the intent references a spending mandate, verifies the delegator's authority is still valid, remaining budget covers the amount, and the mandate hasn't been revoked. Decrements budget atomically via DB row lock.
file: packages/api/src/sardis/routers/mandate_delegation.py SQL: SELECT … FOR UPDATE on mandate row fail-closed: race → NOWAIT lock, 409 to caller
09
Rate limiter + anomaly score.
Per-agent + per-org buckets in Redis. Anomaly model (sardis-guardrails) scores the intent against learned baselines — amount z-score, velocity, new counterparty, unusual time-of-day. Score above threshold routes to approval queue instead of denying.
file: packages/sardis-guardrails/src/sardis_guardrails/rate_limiter.py file: sardis-core/src/sardis_v2_core/anomaly_detection.py fail: over threshold → approval queue, not denial
Stage 4 · Protocol verification

If the request came in under a standard, verify the chain.

10
AP2 / TAP / x402 / A2A mandate chain verification.
If the intent carries a protocol envelope, the matching verifier walks the mandate chain — Ed25519 / ECDSA-P256 sigs on Intent, Cart, Payment mandates; TTL; replay cache lookup in Redis + Postgres archive. An unverifiable chain ends the flow.
file: packages/sardis-protocol/src/sardis_protocol/ap2.py file: sardis_protocol/x402_settlement.py, erc8122.py replay: sardis_protocol/replay_cache_redis.py fail-closed: invalid sig → deny; replay → 409
Stage 5 · Execution

Chain selection, MPC signing, broadcast.

11
ChainExecutor picks chain + token + route.
Reads the request's target chain (default "base"), selects the token (USDC / EURC / PYUSD), and picks a route — direct transfer, CCTP bridge, paymaster-sponsored. Tempo has its own dedicated executor.
file: packages/sardis-chain/src/sardis_chain/smart_router.py file: sardis-chain/src/sardis_v2_chain/executor.py file: sardis_chain/tempo/executor.py
12
WalletManager requests an MPC signature.
sardis-wallet/manager.py hands the payload to the active custody provider (Turnkey default). For Turnkey: organization + sub-org + private key reference. The key never leaves the provider — Sardis receives a signed transaction, not a key.
file: packages/sardis-wallet/src/sardis_wallet/manager.py file: sardis_wallet/tempo_keychain.py, circle_client.py fail-closed: provider down → 503, no broadcast
13
Broadcast via Alchemy RPC.
Signed transaction submitted to SARDIS_BASE_RPC_URL (env-driven, Alchemy primary, public RPC fallback). Tempo uses rpc.tempo.xyz. Executor watches for the receipt with bounded retries.
provider: Alchemy (Base/Ethereum/Polygon/Arbitrum/Optimism) provider: Tempo public RPC fail: receipt timeout → mark pending, reconciler will finalize
Stage 6 · Receipt + audit

Ledger row, Merkle anchor, webhook.

14
Ledger entry + policy decision rows written.
A single transaction writes ledger_entry (the movement) plus policy_decision (why it was allowed) in Postgres. Both rows reference each other. This is the evidence base for disputes and regulator queries.
file: packages/sardis-ledger/src/sardis_ledger/models.py file: sardis_ledger/hybrid.py (ImmuDB optional)
15
Merkle batch → on-chain anchor.
A background job rolls recent ledger entries into a Merkle tree and anchors the root via SardisLedgerAnchor.sol. After anchoring, any ledger row can be proven to have existed at or before a given block.
file: packages/sardis-ledger/src/sardis_ledger/anchor.py contract: contracts/src/SardisLedgerAnchor.sol async: batched, ~5 min cadence in prod
16
Webhook enqueued via outbox; SDK returns receipt.
Response assembled (PaymentResult), returned up the stack. Webhook for payment.succeeded / payment.denied enters the outbox table — delivered at-least-once by the outbox worker with exponential backoff and event-id dedup.
table: webhook_outbox dedup: event_id uniqueness + delivery tracking fail: delivery 5xx → retry with backoff, alert after N failures

What happens when each stage fails.

StageModeObservable effect
Rate limitfail-closed429; if Redis unreachable in prod → 503 (no silent pass-through).
Principal lookupfail-closed401; no orchestrator entry.
Spending policyfail-closed403 with reason code; policy_decision row written.
Compliance (Didit/Elliptic)fail-closed403; override only via SARDIS_AGIT_FAIL_OPEN=true (dev only).
Mandatefail-closed409 on race (NOWAIT); 403 on insufficient budget.
Anomalyfail-softAbove threshold → approval queue, HTTP 202 pending_approval.
