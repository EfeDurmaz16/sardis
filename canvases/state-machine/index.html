Canvas 21 · Protocol

22-State Machine — one Payment Object's full life.

Every Payment Object passes through a deterministic state machine covering the happy path, escrow hold, partial settlement, dispute arbitration, FX routing, and fulfillment acknowledgment. 22 states, each with strict allowed transitions and terminal conditions. This is the most comprehensive payment lifecycle spec designed for agent commerce.

TL;DR Card networks have one happy path and a dispute process bolted on. Sardis treats every possible outcome — direct settle, escrow hold, timelock release, dispute file, arbitration decision, partial settlement, refund, cancel, failure — as a first-class state with explicit transitions. An agent payment never ends up in an undefined condition.

Escrow path · +5 states

When delivery verification matters.

Transactions above the escrow threshold route through an explicit hold. Merchant sees cryptographic proof of locked funds; agent confirms delivery (or timelock auto-releases); dispute routes to arbitration with one of three outcomes.

VERIFIED → ESCROWED → CONFIRMING │ or AUTO_RELEASING → RELEASED → SETTLING → SETTLED → FULFILLED
StateMeaningAllowed next
ESCROWEDFunds in escrow hold (above threshold)CONFIRMING, AUTO_RELEASING, DISPUTING
CONFIRMINGAgent reviewing delivery from merchantRELEASED, DISPUTING
AUTO_RELEASINGTimelock expired without dispute; auto-releasing to merchantRELEASED
RELEASEDEscrow released to merchantSETTLING, DISPUTING (within dispute_window)
PARTIAL_SETTLEDPartial amount settled (less than locked)UNLOCKING, DISPUTED
Default escrow timeouts API/SaaS digital delivery = 1h escrow / 24h dispute / 3-day evidence · Software license = 24h / 7d / 5d · Physical goods = 14d / 30d / 7d · B2B services = 30d / 60d / 14d.

Terminal states · 10

Every object lands somewhere explicit.

Success outcomes
FULFILLED — agent acknowledged service delivery.
RESOLVED_RELEASE — arbitration ruled for merchant.
UNLOCKED — residual FundingCells returned to AVAILABLE.
Failure outcomes
REVOKED — pre-use revocation by principal or issuer.
EXPIRED — expiry time reached before use.
CANCELLED — pre-settlement cancellation → UNLOCKING.
FAILED — processing failure (may retry within mandate.retry_limit).
REFUNDED — post-settlement refund issued.
RESOLVED_REFUND — arbitration ruled for principal.
RESOLVED_SPLIT — arbitration ruled partial.

Why this matters

The "undefined ending" problem.

Card networks
Happy path + dispute chargeback, bolted on. No state for "settled but not delivered." No state for "partial settlement." Merchant and cardholder reconcile via support tickets; evidence is assembled ad hoc; outcomes depend on issuer discretion.
Sardis
Every possible outcome is an explicit state with cryptographic evidence at each transition. An agent running thousands of payments per hour needs deterministic end-states to trigger the right downstream workflow — retry, escalate, refund, reconcile — without human judgment in the loop.
