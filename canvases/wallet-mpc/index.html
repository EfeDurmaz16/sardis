Canvas 09 · Flows

Wallet + MPC

Sardis never holds a private key. Every signature is produced by an external MPC provider — Turnkey by default, Fireblocks / Circle / Lit as swap-ins — behind a single abstract port. This canvas maps the interface, the providers, and the signing path from orchestrator to chain.

TL;DR MPCSignerPort in packages/sardis-chain/src/sardis_chain/executor.py is the contract. TurnkeyMPCSigner, FailoverMPCSigner, and LocalAccountSigner implement it. Wallet lifecycle is owned by WalletManager in sardis-wallet/manager.py. On EVM chains, wallets are Safe Smart Accounts v1.4.1 (pre-deployed). On Tempo, signing is bespoke via TempoKeychainProvider.
Thesis

"Non-custodial" is not a slogan — it's a blast-radius argument.

Keys never live in Sardis memory
Not in env vars, not in Redis, not in a service's process memory. Every signature is a remote call to a provider (Turnkey, Fireblocks, Circle, Lit) that holds the key material in HSMs or threshold-split shards. Sardis sees request → signature, never the key.
Blast radius analysis
If the API server is compromised: attacker gets request metadata, not funds. If the database is exfiltrated: wallet addresses leak, keys don't. If an agent is prompt-injected: the signer still only acts on policy-approved intents. Three independent breaches are required to move money, which is exactly the point.
Pluggable by contract, not by config
The MPCSignerPort interface has three methods. Any provider that implements them is a drop-in. Turnkey was default; Fireblocks/Circle/Lit are integrated; the port makes migration a config change, not a refactor. Lock-in on custody would be the worst kind of lock-in.
Failover is a first-class concern
FailoverMPCSigner wraps an ordered list of providers: primary fails → next. Custody providers occasionally have regional outages; we don't want a Turnkey incident to be a Sardis incident. The port makes this tractable.
Interface

The signing port — three methods.

Any MPC provider that implements MPCSignerPort can drive Sardis. That's the whole contract. Everything else — transaction envelopes, chain selection, ERC-4337 userOp hashing — is handled by the executor before it hits the port. 

class MPCSignerPort(ABC):
    async def sign_transaction(
        self, wallet_id: str, tx: TransactionRequest
    ) -> str: ...                           # returns signed raw tx hex

    async def get_address(
        self, wallet_id: str, chain: str
    ) -> str: ...                           # derives the address for a chain

    async def sign_user_operation_hash(
        self, wallet_id: str, user_op_hash: str
    ) -> str: ...                           # ERC-4337 userOp signature

All three calls happen over the network — they're async by design. The caller (ChainExecutor) never assumes key locality.

Turnkey

The organization model.

Turnkey is multi-tenant by design. Sardis uses that structure as the isolation boundary: one organization per customer / tenant, one sub-organization per agent group, one wallet per agent. Permission grants cascade down; revocation cascades down; nothing cross-cuts.

Config
TurnkeyConfig in packages/sardis-core/src/sardis_v2_core/config.py (env vars: TURNKEY_API_KEY, TURNKEY_ORGANIZATION_ID).
Org per tenant
Root organization is Sardis. Each customer onboarded gets a sub-org with its own API user and quorum rules.
Sub-org per group
Agent groups (agent_groups.py) map to further sub-orgs — keys for one group can't sign for another.
Wallet per agent
Each Wallet row corresponds to a Turnkey wallet; wallet_id is the Turnkey wallet id.
Address derivation
Per-chain via TurnkeyMPCSigner.get_address(wallet_id, chain) — one seed → deterministic addresses for every supported chain.
Stamp auth
Every request is signed with a P-256 "API stamp" — no bearer tokens, no replay. Turnkey rejects stale requests at the edge.
Verification guard
settings.turnkey_configured in config.py must be true in live mode; otherwise chain executor refuses to start.
Lifecycle

From empty slot to retired wallet.

Wallets are stateful: they're created, they sign, sometimes their keys rotate, sometimes they get frozen, eventually some are archived. WalletManager in sardis-wallet/manager.py owns the full lifecycle.

create_turnkey_wallet() / create_circle_wallet(). Row in wallets, wallet record at provider.
02

derive_address(wallet_id, chain). Per-chain address cached in DB.
03

First signing request after policy approval. activity_monitor logs it.

05

07

Wallet retired; history + keys preserved at provider for audit.

Smart accounts

Safe v1.4.1 on every EVM chain.

Agent wallets on EVM are Safe Smart Accounts, not plain EOAs. The singleton is pre-deployed; Sardis deploys a CREATE2 proxy per wallet. This gets us ERC-4337 compatibility, modular policy, and social recovery — without paying audit costs on a bespoke account contract.

Safe4337Module — required for UserOp execution

CREATE2 proxy via Safe's canonical ProxyFactory. Address is predictable before deployment via predict_safe_address() — we can share deposit addresses with users before any gas is spent.
Policy wiring

ERC-4337 stack

Gas

Paymasters — users don't pay ETH.

Asking an agent or merchant to "hold ETH for gas" is a UX dead end. Sardis uses a paymaster on every supported EVM chain so gas is either paid in USDC (Circle) or sponsored (Pimlico).

PaymasterModelClientWhen it's used
Circle Paymaster v0.8 primary User pays gas in USDC. Permissionless, no API key. CirclePaymasterClient in erc4337/paymaster_client.py Default for agent + merchant userOps on Base, Ethereum, Polygon, Arbitrum, Optimism (mainnet + testnet).
Pimlico Sardis sponsors gas (paid out of operating account). PaymasterClient in the same file Fallback when Circle is unavailable, and for flows where the platform picks up gas (onboarding, demos, first-transaction incentives).
Invariant Gas-payment choice is visible in the SubmittedTx metadata and the payment ledger. No silent sponsorship — operators can always tell who paid what.
Tempo

Why Tempo needs its own keychain.

Tempo is EVM-compatible at the execution layer but has a different account/permission model — the Keychain precompile at 0xAAAAAAAA00000000000000000000000000000000 with per-Access-Key spending limits and a custom 86-byte signature format. That's not something a generic ERC-4337 signer handles — it gets its own provider.

Provider
TempoKeychainProvider in packages/sardis-wallet/src/sardis_wallet/tempo_keychain.py
Executor
packages/sardis-chain/src/sardis_chain/tempo/executor.py (dedicated — not shared with EVM executor)
Account helper
packages/sardis-wallet/src/sardis_wallet/tempo_accounts.py
Signature format
0x03 || root_address (20 bytes) || inner_secp256k1_sig (65 bytes) — 86 bytes total. Built by build_keychain_signature().
Access keys
AccessKeyConfig maps Sardis policy directly to on-chain key parameters: token_limits, allowed_targets, max_calls_per_day, expires_at. A Sardis spending policy becomes an on-chain, Tempo-enforced access key.
Root custody
Root key stays in Turnkey. Access keys are short-lived agent keys provisioned per mandate.
Fee payment
packages/sardis-chain/src/sardis_chain/tempo/fee_payer.py — Tempo's native fee-payer abstraction, USDC/pathUSD gas.
Freeze

The wallet-level kill switch.

Freeze is narrower than the global kill switch: it targets a single wallet, not the platform. It's the right response when KYC changes, an incident is localized, or a mandate is revoked — without disrupting the rest of the tenant.

Columns
wallets.is_frozen, frozen_at, frozen_by, freeze_reason — added in migration 006_wallet_freeze.sql.
Enforcement
WalletManager checks is_frozen before dispatching to the signer port. Signers never see the request.
Compliance hook
KYC denial, AML hit, or kill-switch activation can all flip the freeze flag. Every flip is logged with the reason.
Unfreeze
Explicit operator action, audit-logged. No auto-expiry — a freeze stays a freeze until someone decides otherwise.
Shared wallets

Agent groups + multisig.

Not every wallet is for a single agent. Team wallets, treasury wallets, and high-value flows need m-of-n approvals. Sardis supports that at the signer level, not just at the policy level.

Agent groups
packages/sardis-core/src/sardis_v2_core/agent_groups.py. A group has members, group-level policies, and a shared wallet. Turnkey sub-orgs back the grouping so the custody model agrees with the product model.
Multisig signer logic
packages/sardis-wallet/src/sardis_wallet/multisig.py. ApprovalThreshold, MultisigSigner, TransactionApproval — collects the required quorum before the MPC signer is asked to produce a final signature.
Social recovery
setup_social_recovery() / initiate_recovery() on WalletManager. A guardian set can rotate keys if the primary signer is lost — no "the founder died and the keys died with him" failure mode.
Session keys
session_manager.py. Short-lived signing sessions scoped to specific agent flows. Analogous to Tempo Access Keys but at the platform layer for non-Tempo chains.
