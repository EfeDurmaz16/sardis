Canvas 12 · Product

Compliance Stack

The SDK is the distribution. Compliance is the moat. No fintech partner, no regulated merchant, and no enterprise buyer will let an AI agent move real money through them without seeing a credible KYC + AML + sanctions + travel-rule + SAR stack behind the API. Sardis ships one.

TL;DR Six pillars: KYC for humans (Didit), KYB for organizations, KYA for agents, AML + sanctions for addresses and counterparties, Travel Rule for cross-VASP transfers above $3K, and SAR for regulator-bound suspicious activity reports. All of it runs inside the PreExecutionPipeline on the fail-closed side of the gate.
Why this is the moat

Anyone can write a payment SDK. Almost nobody ships compliance.

Every payments-for-agents startup demo looks the same on day one: a wallet, a signed transaction, a receipt. The hard part is day 90 — when a regulated merchant wants to onboard, the bank wants a BSA report, the auditor asks for retention policy, and the OFAC list just got a new entry. These aren't optional; they're the line between a demo and a business.

What partners ask for
Stripe Connect onboarding · Circle business account · Bridge originator · Striga banking — every one of these started with the same question: "show us your KYC / AML / SAR program and your retention policy". Without those, there is no bank rail, no payout, no card issuing. Sardis exists as a platform because the compliance stack exists first.
Why the SDK doesn't replace it
SDKs are distribution — they get Sardis into an agent. They don't move money, don't hold KYC records, and don't satisfy a regulator. The compliance layer is what makes the SDK's output actually spendable. You can't fork your way around FinCEN 5-year retention.
The six pillars

What actually runs.

Each pillar is a concrete module in packages/sardis-compliance/src/sardis_compliance/ with its own provider abstraction, its own persistence, and its own tests.

Pillar 1 · KYC

Know Your Customer — for human users.

Primary provider: Didit (YC W26, $0.33/check, 500 free/mo). Document + liveness + face match. Persona and Coinbase verifications sit behind the same interface. Module: sardis_compliance/kyc.py + provider providers/didit.py. Persistence: kyc_verifications (fixed to TEXT agent_id + kyc_status on users in migration 090).

Pillar 2 · KYB

Know Your Business — for organizations.

Required before an organization can deploy agents that move money above threshold. Module: sardis_compliance/kyb.py. Table: kyb_verifications (migration 010) — inquiry_id, provider, status, verified_at, expires_at, business_name. Triggered at organization onboarding, rechecked on anniversary.

Pillar 3 · KYA

Know Your Agent — the Sardis original.

Human KYC and business KYB don't cover the thing actually signing: the agent. KYA captures agent identity (Ed25519/ECDSA-P256 keys), attestation, heartbeat, and state transitions (register · check · upgrade · suspend · reactivate · stale sweep). Module: sardis_compliance/kya.py. Ties to agit_policy_engine at decision time.

Pillar 4 · AML + Sanctions

OFAC, PEP, adverse media, on-chain.

Providers behind one interface: OFAC, OpenSanctions, Chainalysis, Scorechain, Watchman, Elliptic (planned). Screens addresses (on-chain counterparty), entities (PEP / sanctions lists), and transactions (combined). Module: sanctions.py · pep.py · adverse_media.py. Blocked addresses cached with TTL.

Pillar 5 · Travel Rule

FATF R.16 — originator + beneficiary data.

Migration 012 defines travel_rule_transfers: transfer_id, amount, originator and beneficiary JSONB, protocol (e.g., Notabene). Triggered above $3K USD / $1K EUR. Module: travel_rule.py + providers/notabene.py. Without this, VASP-to-VASP transfers are not legally clean.

Pillar 6 · SAR

Suspicious Activity Reports — FinCEN 5-year retention.

suspicious_activity_reports (migration 011) — sar_id, internal_reference, activity_type, priority, subject, filing_deadline, report_data. Auto-drafted from anomaly scoring; filed by humans. Module: sar.py. Five-year retention is a hard legal constraint; the schema is designed for it.

KYC · Didit flow

Signup to sk_live_ key in four steps.

Router: packages/api/src/sardis/routers/kyc_onboarding.py. Webhook delivery is where a dashboard user becomes a fully provisioned Sardis tenant.

01
Initiate. POST /v1/onboarding/kyc/initiate creates a Didit session (document + liveness). Returns a hosted URL the dashboard redirects to.
02
User verifies. On Didit's hosted UI. No PII crosses Sardis at this step.
03
Webhook. POST /v1/onboarding/kyc/webhook — Didit posts the decision with x-api-key auth. Row inserted into kyc_verifications; users.kyc_status flipped to verified / denied.
04
Live key issued. On verified, the user becomes eligible for a production API key (sk_live_...). Before that, only sk_test_ sandbox keys are issued. Retry endpoint: POST /v1/onboarding/kyc/retry.
At execution time

What actually runs inside the Pre-Execution Pipeline.

The orchestrator calls the pipeline once. The pipeline runs the compliance checks in the order below. Any fail-closed reject is audited with a reason code and surfaced via webhook. 

sequenceDiagram
    participant O as Orchestrator
    participant P as PreExecutionPipeline
    participant KYC as KYC (Didit)
    participant KYA as KYA (agent)
    participant S as Sanctions (OFAC / Chainalysis)
    participant F as FIDES (trust graph)
    participant TR as Travel Rule (Notabene)
    participant SAR as SAR drafter
    participant L as Ledger

    O->>P: execute(intent)
    P->>KYC: user.kyc_status == verified?
    KYC-->>P: ok / denied
    P->>KYA: agent registered + active?
    KYA-->>P: ok / stale / suspended
    P->>S: screen(payee, payer, amount)
    S-->>P: clear / match / soft-match
    P->>F: trust_score(agent_did) ≥ threshold?
    F-->>P: tier + signals
    alt amount > $3K
      P->>TR: originator/beneficiary ready?
      TR-->>P: drafted / blocked
    end
    alt any anomaly signal
      P->>SAR: draft report
      SAR-->>P: queued for human filing
    end
    P->>L: log decision (approved | denied)
    P-->>O: verdict

Agent-native additions

FIDES + KYA — what human compliance frameworks can't cover.

KYC/AML/Travel Rule were designed for humans and companies. Autonomous agents break both axes — they don't have passports and they aren't LLCs. Sardis fills the gap with two modules.

FIDES — decentralized trust graph
Migration 055 adds did_mappings, trust_score_snapshots, and agents.fides_did / agents.agit_repo_hash columns. An agent's DID is verified via signature; its trust score is recomputed from on-chain behavior, incident history, and cross-platform attestations. Router: routers/fides_identity.py. This is Efe's other open-standards project, embedded as a first-class compliance signal.
KYA — agent lifecycle
Eleven endpoints under /v1/kya/*: register, get, check, upgrade, suspend, reactivate, heartbeat, stale-sweep. Module: sardis_compliance/kya.py. agit_policy_engine.py consumes KYA state at decision time — a suspended or heartbeat-stale agent fails closed before the transaction reaches the wallet.
Data governance + enterprise controls

Retention, export, SSO, and per-role RBAC.

Compliance doesn't end when the check passes. Regulators want retention schedules, GDPR wants export + deletion, and enterprise buyers want SSO before they write a PO.
