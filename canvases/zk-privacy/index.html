Canvas 22 · Extended

ZK Privacy Stack

Prove a payment satisfied its policy, mandate, funding, and identity requirements — without revealing the amounts, the limits, the merchant, or the agent. Sardis ships two Noir circuit packages (sardis-zk-policy, sardis-zkp), a deterministic Python simulator for CI, and three adjacent runtime layers — sardis-ucp, sardis-guardrails, and the PreExecutionPipeline — that decide when a clear-text decision needs to turn into a proof.

TL;DR Enterprises can't publish raw spending logs to auditors — the logs are the product. ZK lets the same row say "this payment was within all 11 policy rules" without showing per-transaction amounts, merchant identities, or the limits that define the rules. Sardis uses Noir + Barretenberg (Aztec) for the circuits, because the toolchain is audited, the proving backend is fast enough for per-transaction use, and verification is cheap enough to run on-chain via SardisLedgerAnchor.
Status

What ships vs. what's experimental.

The ZK stack is real code — two Noir projects, working circuits, a Python simulator used in CI. It is not on the default hot path. Today it runs as an alternative policy check for tenants that explicitly opt in; the clear-text PreExecutionPipeline remains the production default.

Backend Choice

Why Noir (Aztec) — and not Circom / Halo2 / RISC0.

Rust-like syntax, small surface
Circuit authors are Python and Rust engineers, not cryptographers. Noir reads like Rust — assert, for, if, u64 — which keeps the policy circuit reviewable alongside the clear-text Python implementation in policy_circuit_sim.py.
ACIR → ACVM → Barretenberg
Noir compiles to ACIR (intermediate representation), executed by the ACVM witness solver, and proven by Barretenberg (UltraPlonk + Honk). That layering means we can swap proving backends later without rewriting circuits.
Audited toolchain
Aztec audits its compiler and prover. The alternative — hand-rolled Circom — requires writing templates + solvers + verifiers that all need independent review. For a fintech, using an audited toolchain is cheaper than auditing custom R1CS.
Cheap on-chain verification
Barretenberg emits Solidity verifier contracts. One ZK policy proof verifies in ~300K gas on Base L2 — low enough to anchor alongside the Merkle batch from SardisLedgerAnchor.sol without blowing the per-batch cost model.
Package · sardis-zk-policy

Policy circuit — 5 of 11 checks in zero knowledge.

A single Noir binary at packages/sardis-zk-policy/src/main.nr. Private inputs carry the transaction + policy parameters; a single public input pair (evidence_hash_high, evidence_hash_low) links the proof to the off-chain policy decision log.

Package · sardis-zkp

Reusable circuit library — mandate, identity, funding.

Three Noir circuits under packages/sardis-zkp/circuits/, orchestrated by a Python ZKProver in src/lib.py. Each circuit targets one question; together they cover the questions Sardis actually needs to answer in ZK at the mandate layer.

mandate_compliance.nr — payment within mandate bounds
packages/sardis-zkp/circuits/mandate_compliance.nr
Proves amount ≤ per_tx_limit AND daily_spent + amount ≤ daily_limit, and that the private inputs hash to the public mandate_commitment and payment_commitment (both Poseidon on BN254). Merchant inclusion via 8-deep Merkle proof; replay protection via mandate_nonce.

Privacy Tiers

Transparent, hybrid, full ZK.

Defined in sardis_zkp.PrivacyTier. Tenants select a tier per payment object (mandate, payment, wallet) — the orchestrator routes to the corresponding verifier path.

TierWhat's visibleWhat's hiddenWhen to use
TRANSPARENTAll fields (amount, merchant, limits, counterparty)NothingDefault. Consumer agents, developer sandboxes, public tokens.
HYBRIDMerchant, chain, timestamp, decision outcomeAmount, limitsB2B payments where merchant identity is non-sensitive but commercial terms are.
FULL_ZKProof only — commitments + verification resultAmount, merchant, limits, agent identity, funding cellsEnterprise treasury flows, regulated payouts, custodial agent fleets.
Coverage Matrix

Which policy checks translate to ZK — and which don't.

The SpendingPolicy engine has ~11 check types. Five are implemented in main.nr today. The rest either need oracle data, a running graph, or third-party attestations that can't be proven in-circuit without leaking the external state.

Policy checkZK todayZK feasibleWhy / why not
Per-transaction capYes—Pure numeric comparison.
Daily aggregate capYes—Running total carried as private input.
Per-merchant capYes—Keyed by merchant_id_hash; same shape as daily.
MCC block-listYes (8 slots)—Set-membership via linear scan; extend to Merkle for larger lists.
Time window (with wrap)Yes—Hour as u8; handled in both linear and wrap case.
Allowed-merchant Merkle rootPartial (in sardis-zkp mandate)YesNeed to wire merchant registry → Merkle root into prover input.
KYC/AML statusNoYes, via attested credentialRequires a KYC-provider-signed attestation usable inside circuit.
Sanctions / OFAC screenNoPartialElliptic API is external; proof would cover attestation, not live screen.
Anomaly score thresholdNoPartialScore is an ML output; circuit could assert a signed score, not compute it.
Rate limit (token bucket)NoYes, with state commitmentNeeds a commitment to bucket state; opens replay questions.
Kill switch / emergency stopNoNoKill switch is operator intent, not a math fact. Stays clear-text.
Flow

Prover → verifier, end to end.

A ZK-tier payment takes the same orchestrator spine as a transparent one — it just forks at the policy check into a proof-generation path instead of a direct assert. 

sequenceDiagram
    participant A as Agent
    participant O as PaymentOrchestrator
    participant P as PreExecutionPipeline
    participant ZP as ZKProver (sardis-zkp)
    participant N as Noir / nargo
    participant L as Ledger + Anchor
    participant V as On-chain Verifier

    A->>O: execute(intent, privacy_tier=FULL_ZK)
    O->>P: run classical gates (rate limit, kill switch)
    P->>ZP: prove_mandate_compliance(amount, limits)
    ZP->>N: write Prover.toml + nargo prove
    N-->>ZP: proof bytes + public inputs
    ZP->>P: ZKProof(commitments, bytes)
    P->>P: hash evidence row → evidence_hash
    P->>ZP: prove_policy(evidence_hash, witness)
    ZP-->>P: policy proof bytes
    P-->>O: approved (ZK)
    O->>L: append row + attach proofs
    L->>V: submit Merkle root + ZK proof batch
    V-->>L: on-chain receipt

Adjacent · sardis-ucp

Universal Commerce Protocol — the object model ZK wraps.

UCP is not ZK, but it defines the commerce primitives (UCPCartMandate, UCPPaymentMandate, UCPOrder, UCPCheckoutCapability) that ZK proofs reference as commitment inputs. When mandate_compliance.nr takes a payment_commitment, the preimage is a UCP mandate object.

Adjacent · sardis-guardrails

Classical runtime safety — always in front of the proof.

Guardrails are not ZK; they run first. A proof that a payment satisfied a policy is worthless if the agent is on the kill list or the circuit breaker is tripped. Guardrails answer "should we even be computing a proof right now?"

Integration

Where ZK hooks into the spine.

The PreExecutionPipeline runs all transactions through the same ordered gate (see canvas 01 · Architecture). The ZK path is a fork inside the policy gate, not a parallel pipeline — everything else (mandate check, AP2 verify, rate limit, kill switch) is shared with the transparent path.

What the orchestrator does differently
On privacy_tier=FULL_ZK, the policy check returns a ZKProof instead of a boolean. The orchestrator attaches the proof to the ledger row, and the Merkle batch submission includes proof bytes for on-chain verification.
What stays the same
Kill switch, rate limit, input validation, MPC signing path, CCTP settlement, reconciliation queue. ZK changes what is seen; it doesn't change what is done.
Dev vs. prod mode
ZKProver(production=False) emits JSON placeholder proof bytes and refuses to verify. production=True shells out to nargo prove / nargo verify in the circuit directory. Mock verify was removed on purpose to stop anyone from mistaking dev mode for security.
On-chain anchor
Verifier contracts generated by Barretenberg are deployed alongside SardisLedgerAnchor.sol (canvas 06). A batch root + proof tuple posted there is independently verifiable by any third party without reading Sardis logs.
