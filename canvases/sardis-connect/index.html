Sardis Connect

Make any API agent-ready in three lines of code. The merchant ships USD, the agent pays in stablecoins, Sardis handles every crypto primitive in between. This canvas explains what Connect is, how it differs from Pay with Sardis, and how the full onboarding flow works end-to-end.

TL;DR Pay with Sardis is a customer-facing checkout (buyer clicks "Pay with Sardis" on a merchant site). Sardis Connect is the merchant-facing inverse — a drop-in SDK that turns any HTTP API into an agent-payable service: discovery, payment, verification, and USD settlement. Merchant integrates in 5 minutes, never touches a wallet, receives USD in their connected bank account.
Audience

Who actually ships Sardis Connect.

Connect is a B2B2Agent wedge. The merchant is any company that already runs an HTTP API and wants AI agents as a new revenue channel — without learning what a gas fee is.

AI API providers
Text-generation, embeddings, analysis, transcription, image, speech APIs. Today they gatekeep with API keys + human-signed Stripe subscriptions. With Connect, every endpoint gets a price and an agent can pay per call — no signup, no CAPTCHA, no credit card form.
Data and scraping APIs
The long tail of priced HTTP APIs (proxies, weather, FX, company data). The per-request monetization loop is the native Connect shape: agent discovers via /.well-known/sardis.json, pays $0.001 per call, uses it.
SaaS platforms exposing agent features
A traditional SaaS (CRM, analytics, ticketing) adds Connect to bill agents for programmatic access. Human users stay on the existing subscription plan; agents get a per-call plan without any pricing-page gymnastics.
Services in the OSP ecosystem
A provider implementing Open Service Protocol (OSP) can declare sardis_wallet in accepted_payment_methods. Connect is the drop-in library that makes that declaration real — payment, verification, settlement. See Canvas 21 · OSP Integration.
The Flow

Merchant onboarding, end to end.

Four moving parts: the merchant app, the Connect SDK, the Sardis API (merchant checkout), and the merchant's connected bank rail. The merchant touches three of the four; Sardis orchestrates the fourth.

Python SDK

Three lines in FastAPI.

Straight from packages/sardis-connect/README.md. The router exposes three endpoints: agent discovery, payment session creation, and payment verification. 

from fastapi import FastAPI
from sardis_connect import SardisConnect

app = FastAPI()

# 1. Initialize with your merchant credentials
sardis = SardisConnect(
    api_key="mch_live_xxx",
    merchant_id="merch_xxx",
    service_name="My AI API",
)

# 2. Define priced endpoints (or sardis.meter() for per-unit)
sardis.price("/api/generate", amount="0.05", description="Generate text")
sardis.price("/api/analyze", amount="0.10", description="Analyze data")

# 3. Mount the router
app.include_router(sardis.router)
EndpointMethodPurpose
/.well-known/sardis.jsonGETAgent discovery manifest — name, accepted rails, endpoints, prices.
/sardis/payPOSTCreate a merchant-checkout session; returns session_id + client_secret.
/sardis/verifyPOSTLook up session status; returns { verified, amount, payer_id }.
/sardis/webhooksPOSTSigned (HMAC-SHA256) webhook receiver for payment events.
/sardis/usagePOSTReport metered usage (tokens/units) for per-unit pricing — see SardisConnect.meter().

Protect any endpoint with the SardisPaywall FastAPI dependency — the client passes X-Sardis-Session: mcs_… and the dependency verifies status against the Sardis API before the handler runs. Source: packages/sardis-connect/src/sardis_connect/middleware.py.

Node SDK

Express / Node.js, same three lines.

Published as @sardis/connect. Source in packages/sardis-connect-js/src/middleware.ts. 

import express from "express";
import { sardisConnect } from "@sardis/connect";

const app = express();

const sardis = sardisConnect({
  apiKey: "mch_live_xxx",
  merchantId: "merch_xxx",
  serviceName: "My AI API",
  baseUrl: "https://api.example.com",
});

sardis.price("/api/generate", "0.05", "Generate text");
sardis.price("/api/analyze", "0.10", "Analyze data");

app.use(sardis.middleware());

The middleware hands off to next() for anything that isn't a Sardis route, so it composes cleanly with existing Express apps. The manifest declares accepts: ["sardis", "x402", "mpp"] — agents can choose whichever rail they prefer; the server-side flow is identical.

Reference Demo

The E2E script, step by step.

demos/sardis_connect_e2e_demo.py walks the full lifecycle in ~320 lines. Each step maps to one subsystem.

1
Merchant registration
Merchant(name, settlement_preference="stripe_connect", mcc_code="5734"). Produces a merchant_id and an API key in the mch_live_… namespace. Backed by sardis-core/merchant.py and the merchants router.
2
Merchant onboarding
Hosted Stripe Express flow (KYB + bank details). On return: charges_enabled, payouts_enabled, onboarding_state=complete. The merchant record is stamped with stripe_account_id; this is where USD will land.
3
Agent discovery
Agent does GET /.well-known/sardis.json. The manifest is built by SardisConnect._get_manifest() from the in-process endpoint registry plus accepts: ["sardis", "x402", "mpp"].
4
Spending mandate check
Before calling /sardis/pay, the agent's SpendingMandate is evaluated: per-tx, daily, monthly caps, merchant_scope allowlist, allowed_rails. Denials include a machine-readable reason code. See Canvas 08 · Policy Engine.
5
Natural-language policy
The demo parses phrases like "Maximum $50 per day on AI API calls" via RegexPolicyParser in sardis-core/nl_policy_parser.py. The parsed structure becomes a signed mandate.
6
Multi-protocol execution
Same agent, same mandate, three rails: direct USDC (Base/Tempo), x402 (HTTP 402 on micropayments), and MPP (Stripe session). Connect is rail-agnostic because the manifest advertises all three and the Sardis orchestrator selects the best fit.
7
Settlement to USD
Sardis settlement engine fires a transfer to acct_…; Stripe's auto-payout pushes USD to the merchant's bank. Reconciliation enqueued in the Postgres reconciliation queue (see Canvas 10 · Settlement & Chains).
Settlement Provider

Why the merchant never touches crypto.

Why a bank-connected merchant account
The merchant already trusts Stripe for card payments. Connect's Express account model amortizes KYB/KYC across their existing compliance stack, so Sardis does not rebuild it. The tradeoff is Stripe's fee on the USD leg — Sardis eats none of that margin on the crypto leg (gasless via Circle Paymaster, USDC native).
Alternative rails
EU merchants can opt into Striga (SEPA + vIBANs). Multi-rail merchants can use Bridge (ACH/SEPA/PIX). Lightspark is wired for Lightning. The settlement_preference field on the Merchant record is the toggle — abstracted behind sardis-core/settlement.
Economics

Pricing + settlement model.

MechanismHow it worksWhen to use
Fixed pricesardis.price(path, amount) — agent pays the posted price per call.Predictable endpoints: generation, embeddings, search.
Metered (per-unit)sardis.meter(path, per_unit, unit) — agent calls, merchant reports units via /sardis/usage, Sardis computes charge.LLM completions (tokens), vision (pixels), data egress (bytes).
Platform feeSardis takes a flat percentage on the USDC leg. Stored as platform_fee_amount on each session. Breakdown exposed to both sides.How Sardis monetizes. Transparent on every receipt.
Merchant payoutTransfer to merchant bank → auto-payout; configurable schedule (daily/weekly).US default. Swap to Striga/Bridge for EU/multi-rail.
Two Products, One Rail

Sardis Connect vs. Pay with Sardis.

Both sit on the same merchant-checkout session primitive, but the audience and ergonomics diverge. Don't conflate them in sales conversations.
