Canvas 19 · Ops

Security Model

Sardis handles money on behalf of AI agents that can be prompt-injected, API keys that can leak, and merchants that may be malicious. The security model is a chain of small, auditable decisions — every one of them fail-closed — anchored by an append-only ledger so nobody, including us, can quietly edit history.

TL;DR No private keys in Sardis memory. No silent fallbacks. Every payment traverses the PreExecutionPipeline before a signature is requested. Every decision is written to policy_decisions and anchored on-chain. One kill switch freezes every wallet in under a second.
Trust Boundaries

Who trusts whom, and for what.

Threat model

Who breaks in, and what stops them.

Each row is an actor we actively design against. The "mitigations" column is code that already exists — not a wishlist.

ActorAttackMitigationsResidual risk
Compromised agent Prompt injection tricks the agent into calling pay() with a hostile recipient / amount. Non-custodial (no keys to steal), spending policy caps per-txn and per-window, AGIT fail-closed, anomaly scorer (anomaly_detection.py), human approval for risky actions (approval_service.py). Agent can drain up to policy cap. Mitigation: tight caps + approval thresholds.
Compromised API key Key leaks via logs, model output, or client-side reverse-engineer. Keys hashed SHA-256 at rest, rate limiter fail-closed if Redis unreachable, anomaly detector flags new-geo usage, kill switch freezes wallets in one call. Small window before anomaly fires. Mitigation: customer rotation workflow + audit alerts.
Compromised MPC provider Attacker gains Turnkey org-level access. Sub-org per tenant — blast radius capped at one customer. Turnkey policy engine enforces the same spending rules server-side. Catastrophic if root org compromised. Mitigation: provider abstraction lets us add Fireblocks/Circle/Lit as a second root.
Malicious merchant Merchant fakes agent signals to claim funds, or sets up checkout to phish users. Merchant trust scoring, capability gating, embed-origin CORS restriction (embed_origin column, migration 031), session client_secret + idempotency_key unique constraint, SELECT … FOR UPDATE NOWAIT on settlement. Policy-level abuse (false advertising etc.). Mitigation: legal review + offboarding pipeline.
Insider (Sardis employee) Staff modifies a ledger entry to hide a bad decision. Append-only ledger_entries, policy_decisions writes per request, Merkle-anchored batches via SardisLedgerAnchor.sol, evidence export signed by chain state. A re-anchor could theoretically rewrite recent history if insider controls the deployer key — but divergence is visible on-chain.
Replay / forgery Attacker re-submits a valid mandate to execute again. Postgres + Redis replay cache (MandateArchive + dedup_store.py), nonce enforcement, DB-level idempotency_key unique constraint (migration 031), mandate TTLs. Low — doubled on two independent stores.
Provider outage Didit / Elliptic / Turnkey down — tempts "just approve for now." Every pipeline step defaults to deny on timeout or error. No "allow on failure" path in prod. Availability hit — accepted trade-off against silent compromise.

Secrets + transport

Secrets management and header posture.

Secrets never in code
All provider credentials live in env vars loaded from Google Secret Manager or Vercel env. gitleaks runs on every PR with full history (secret-scan.yml, fetch-depth: 0). A matched leak fails the run — no merge. API keys are stored in Postgres as SHA-256 hashes, not plaintext.
Middleware stack (sardis-api)
Order from packages/sardis-api/src/sardis_api/main.py: RequestId → StructuredLogging → SecurityHeaders (CSP/HSTS) → RequestBodyLimit (10MB default, 1MB webhooks, 512KB checkout, 64KB policies) → RateLimit → TAP → x402 → UsageMetering → ActivityLogger → CORS (no wildcards — settings.allowed_origins_list) → routers. Every step is fail-closed.
Webhook signatures
Required in all non-dev environments. HMAC via X-Sardis-Signature and X-Sardis-Timestamp. Delivery uses the transactional outbox pattern — webhook_outbox (migration 032) is written in the same transaction as the business operation and flushed by a background worker. event_id keyed dedup.
Replay + idempotency
Redis + Postgres replay caches for mandates (Phase 3 durable state). idempotency_key DB-level unique constraint on checkout sessions (migration 031). SELECT … FOR UPDATE NOWAIT prevents double-pay race at settlement.
Hardening history

TDD remediation — three phases, March 2026.

The security posture above is the result of a deliberate three-phase hardening done in March 2026. Each phase was merged incrementally with tests. Design doc: docs/plans/2026-03-09-tdd-remediation-90-day-design.md.

PhaseFocusWhat shipped
Phase 1 Security + correctness OAuth CSRF fix, admin fail-closed defaults, webhook signatures required in prod, gitleaks in CI, group-policy TTL + budget decrement, queue interface fix, AGIT default fail-closed.
Phase 2 Single execution path PreExecutionPipeline built; AGIT/KYA/FIDES hooks; router migrations (wallets, mandates, ap2) onto PaymentOrchestrator; DI lockdown (chain_executor private); unified PaymentResult.
Phase 3 Durable state + moat Redis dedup store, Postgres reconciliation queue, Redis replay cache, rate limiter enforcement, attestation envelope endpoint, README maturity tiers.
Incident response

Kill switch.

If we see a confirmed compromise, one privileged call freezes every wallet. The action is itself audited — see migration 064_emergency_freeze.sql.

What gets frozen
All MPC signing requests denied at the orchestrator boundary, in-flight batches cancelled before broadcast, background reconciliation suspended. Chain-side funds aren't moved — they simply stop being signable from Sardis. Non-custodial works in our favour here.
Audit record
emergency_freeze_events: action (freeze_all / unfreeze_all), triggered_by, wallets_affected, reason, notes, created_at. Indexed descending on created_at so the operator dashboard can always show the latest action first.
