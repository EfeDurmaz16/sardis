Canvas 08 · Flows

Policy Engine

The hard part of agent payments isn't access to money — it's controlling authority over money. The policy engine is Sardis's answer: natural-language intent → structured rule → enforced on every transaction, logged with a reason code, anchored on-chain. This canvas walks the whole pipeline.

TL;DR SpendingPolicy.evaluate() in packages/sardis-core/src/sardis_v2_core/spending_policy.py runs twelve checks in a fixed order; the first failure short-circuits with a reason code. Above it sits a natural-language parser (Groq Llama 3.3 70B, regex fallback). Below it sits an approval queue, a decision log, a simulation sandbox, and — for mandates that clear the gate — on-chain enforcement via ERC-8183.
Thesis

Why "authority" — not "access" — is the problem.

Giving an agent a wallet takes ten minutes. Giving an agent a wallet it cannot misuse is the actual product. Every design choice in this engine follows from that.

Access is easy. Revocation is hard.
An MPC signer solves "the agent can pay." It does nothing about "the agent decided to pay $40K to a phishing address at 3am because a jailbreak prompt told it to." Policy is the layer that says no on the agent's behalf, before the signer is even asked.
Authority must be structured, not trusted.
Natural language ("$500/day on API credits, openai.com only") is the input, not the enforcement. The parser converts intent into a typed SpendingPolicy dataclass with hard caps, allowlists, time windows — data structures that can be evaluated in <1ms on every tx.
Fail-closed, with a reason.
Every denial returns a machine-readable reason code (per_transaction_limit, merchant_denied, goal_drift_exceeded). No silent rejects, no vague "declined." Agents, operators, and regulators see the same root cause.
Evidence beats logs.
Every allow and every deny writes a row to the decision log. Batches are Merkle-rooted and anchored by SardisLedgerAnchor.sol. "Trust our logs" is not a defense; a Merkle proof is.
Pipeline

Natural language → structured policy.

Operators type English. Agents hit a typed dataclass. The parser bridges the two and is implemented twice — once as an LLM call, once as a regex fallback — so the stack never hard-fails on a missing API key.

Input (what the operator types)
Spend up to $500/day on API credits,
only to api.openai.com and api.anthropic.com.
Require human approval above $200.
Block gambling (MCC 7995) and adult (MCC 5967).
Output (what the engine sees)
SpendingPolicy( agent_id = "agent_123", trust_level = TrustLevel.MEDIUM, limit_per_tx = Decimal("200"), daily_limit = TimeWindowLimit( limit = Decimal("500"), window = "daily", scope = SpendingScope.COMPUTE, ), merchant_rules = [ MerchantRule(merchant_id="api.openai.com", allowed=True), MerchantRule(merchant_id="api.anthropic.com", allowed=True), ], blocked_merchant_categories = ["gambling", "adult"], approval_threshold = Decimal("200"), )
Primary: Groq Llama 3.3 70B
NLPolicyParser in nl_policy_parser.py. Uses llama-3.3-70b-versatile via the Groq OpenAI-compatible endpoint (https://api.groq.com/openai/v1). ~200ms typical; open-source model so there's no proprietary lock-in on the parsing layer itself. Falls through to OpenAI if GROQ_API_KEY is unset and OPENAI_API_KEY is present.
Fallback: regex parser
RegexPolicyParser in the same file. Pattern-matches common phrasings ("$X/day", "only to Y", "block Z") so policy creation still works in airgapped / sandbox environments with no LLM key. Less expressive, but never hallucinates a field. Invariants live in ImmutableParserHardLimits — hard ceilings the LLM can't raise.

Evaluation

Twelve checks, fixed order, first fail wins.

The order in SpendingPolicy.evaluate() is not arbitrary — cheap local checks run first, DB and on-chain lookups later. Short-circuiting saves latency on the common "just blatantly over the limit" case.

Amount validation
amount > 0 and fee >= 0. Reason codes: amount_must_be_positive, fee_must_be_non_negative.
Scope allowed
scope ∈ allowed_scopes (or ALL). Reason: scope_not_allowed.
MCC not blocked
_check_mcc_policy(mcc_code) — 4-digit category screen.
Per-transaction limit
Effective cap = min(category override, trust-tier cap, global cap). Reason: per_transaction_limit.
Lifetime cap
From policy_store in prod, in-memory in dev. Reason: total_limit_exceeded.
Time-window limits
Daily / weekly / monthly rolling windows. Reasons: daily_limit_exceeded, weekly_limit_exceeded, monthly_limit_exceeded.
On-chain balance
If rpc_client provided — is the wallet actually fundable? Reason: insufficient_balance.
Merchant rules
Allowlist, blocklist, per-merchant caps. Reason: merchant_denied.
Goal drift
drift_score > max_drift_score → deny. Reason: goal_drift_exceeded.
Merchant trust
First-seen or low-trust merchants fail unless the policy explicitly allows unknown destinations.
Approval threshold
If amount ≥ approval_threshold: return (True, "requires_approval") and hand off to approval queue.
KYA attestation
For high-value paths, verify on-chain agent identity via kya_client. No attestation → deny.

Reason codes

Every denial is explainable.

These strings are the contract between engine, operator dashboard, and the decision log. Don't invent new ones casually — they're indexed, alerted on, and quoted in incident reports.

On-chain enforcement

Spending mandates — from a Python object to an EVM guard.

A policy is off-chain: Sardis trusts itself. A spending mandate is the same policy cryptographically committed — first to Sardis's append-only log via AGIT, then to an on-chain guard via ERC-8183. If Sardis is compromised, the chain still enforces the cap.

Off-chain: AGIT policy chain
AgitPolicyEngine in agit_policy_engine.py. Every commit_policy() hashes the policy and links it to the prior commit, forming a tamper-evident chain. verify_policy_chain(agent_id) proves the current policy descends from the original — a policy flip doesn't go unnoticed.
On-chain: ERC-8183 guard
ERC8183PolicyGuard in erc8183_policy_guard.py. Before funding or settlement, the guard calls evaluate_funding / submit_funding / submit_settlement — the solidity-side hooks reject if the on-chain policy record doesn't match. Caps currently enforced: 1% fee ceiling, USDC-only, trust hooks disabled (post-audit relaxation path).
Why both?
Off-chain policy runs on every request — cheap, flexible, covers all rails including fiat. On-chain policy is the last line when counterparty trust in Sardis drops to zero (regulator, enterprise treasury, dispute). They agree in the common case; they diverge only when Sardis itself is the attacker, and the on-chain side wins.
ERC-20 hook surface
The guard publishes three evaluation points on-chain: evaluate_funding (pre-transfer sanity), submit_funding (commits the intent to a job), submit_settlement (finalizes the settlement hash). Each raises ERC8183PolicyDenied on mismatch.
Delegation

Parent → child mandate trees.

Real agent systems are multi-agent. A team lead agent hires helpers; a workflow fans out to sub-tasks. Each child needs spending authority — but strictly less than the parent's, never more. That's what mandate_tree.py enforces.

MandateTreeNode
A node holds the mandate (limits, scopes, expiry) plus a list of children. is_leaf() and total_delegated() let callers reason about the tree without walking it manually.
MandateTreeValidator.validate_delegation()
Invariants: child limit_per_tx ≤ parent's; child total_delegated ≤ parent.limit_total; child expiry ≤ parent expiry; scope must be a subset. Violation → deny. This is the "can't issue more authority than you have" rule, enforced structurally.
build_mandate_tree()
Reconstructs the full tree from the DB for audit and visualization. The dashboard's mandate view reads from this; investigators reading a decision log can reconstruct the delegation chain that authorized a specific payment.
Revocation cascades
Revoking a parent mandate invalidates every descendant at the next evaluation. No "zombie sub-agents kept spending after the main agent was killed" story. This matters for incident response.
Approvals

When policy says "maybe."

A request above approval_threshold isn't denied — it's queued. A human operator, a second agent, or a group vote can release it. ApprovalService in approval_service.py owns the lifecycle.

MethodWhat it does
create_approval()Opens an approval record tied to the pending mandate. Notifies via ApprovalNotifier (Slack / email / webhook).
approve(approval_id, approver_id)Marks approved; orchestrator resumes execution on next poll or event.
deny(approval_id, reason)Hard deny — decision log captures the human reason. The mandate is archived, not executable.
cancel(approval_id)Withdrawn by the agent (e.g. order was rescinded upstream).
expire_pending()Cron-called. Approvals have a TTL — stale requests don't execute days later when no one remembers.
list_pending()Powers the operator dashboard's approval inbox.
Decision log

Every allow, every deny, one row.

The policy decision log is separate from the payment ledger — it captures policy outcomes whether or not a payment ever executed. A denied request still has to be evidenced: that's how the agent owner proves to a regulator that the system was working, not failing.

Table
policy_decisions (see canvas 05 · Database).
Rows written when
Every call into SpendingPolicy.evaluate(), plus every approval transition.
Fields
agent_id, mandate_id, reason_code, amount, merchant, chain, token, decided_at, approver_id (if any), evidence_hash.
Anchoring
Batched into Merkle roots by sardis-ledger, anchored via SardisLedgerAnchor.sol.
Export
evidence_export router produces signed, proof-bearing CSV/JSON for audits.
Simulation

Test a policy before it enforces anything.

Policy changes are risky: tighten too much, block legitimate spending; loosen too much, invite incidents. The policy_simulation router (packages/api/src/sardis_api/routers/policy_simulation.py) replays historical traffic against a proposed policy and returns what would have changed — allows that became denies, denies that became allows, new approval routes.

Replay
Takes a time range of real mandates, runs them through the draft policy without touching wallets or chains. Output: diff of outcomes, broken down by reason code.
What-if
Operator uploads a synthetic batch ("ten $150 API spends, two $2K one-offs"). Engine returns the full evaluation trail — same twelve checks, no side effects.
Incident response

Kill switch — the big red button.

When something is clearly wrong ("every agent is suddenly trying to pay the same new address"), the operator needs one action that stops everything. The emergency router is that action.

Router
packages/api/src/sardis_api/routers/emergency.py
Dependency
packages/api/src/sardis_api/kill_switch_dep.py — enforced on every write path.
Effect
Global flag flips; SpendingPolicy.evaluate() short-circuits to deny with reason kill_switch_active; no new mandates accept; in-flight signatures are aborted at the wallet manager.
Scope modes
Global, per-tenant, per-agent, per-chain — you don't always want to freeze everyone.
Audit
Activation, reason, operator identity, and deactivation all log to the decision stream.
Recovery
Manual — intentionally. Restoring spending after a kill requires explicit operator action, not a timeout.
