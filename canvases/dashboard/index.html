Canvas 16 · Product

Dashboard UI

Forty-three operator pages under apps/dashboard/. Next.js 16 App Router (React 19), Tailwind v4, shadcn/ui, monokrom grayscale theme. Built for finance and engineering operators to see, approve, and kill agent spend without a support ticket — and nothing on the page competes with the data.

TL;DR The dashboard is the control plane. It does not sign transactions and it does not hold private keys. It reads from the same Postgres the API writes to, streams live events over SSE, and writes back via the same REST surface that SDKs use — with a human in front of every action.
Stack

What's running under apps/dashboard/.

Every dependency here is in apps/dashboard/package.json. No component is custom-rolled when shadcn already has one; no icon set outside Phosphor; no gradients; no emoji.

Route Structure

Two kinds of surface: the protected operator app and public flows.

(dashboard) is a Next.js route group — it applies a shared layout (auth guard + sidebar + top bar) without appearing in the URL. /demo, /auth/*, and /account/* live outside the group so they render without the operator chrome.

Design Language

Monokrom, tabular, keyboard-first.

Everything grayscale except rare risk-signal accents (policy-denied red on action; never on decoration). Phosphor regular weight, 16/20px. Numerals tabular; sparse grid. A page like (overview)/page.tsx is a stack of shadcn Card blocks — no dashboards-inside-dashboards, no widget marketplace.

Themes
next-themes drives light/dark; both share information density and contrast. No third "brand" theme. Toggle lives in the user menu.
Charts
recharts with a single grayscale palette. No pie charts. No background gradients. Axis labels in Geist Mono; tabular nums for values.
Keyboard
cmdk command palette (⌘K) jumps anywhere; every table supports arrow-key row navigation; destructive actions always require confirm-with-type.
Icons & forbidden lists
Phosphor only. No emoji anywhere in-product. No gradients. No drop shadows deeper than shadcn's default. No custom one-off components when a primitive already exists.

Auth

better-auth + legacy JWT, same session.

better-auth handles the dashboard side (sessions, passkeys, roles, API-key plugin, Polar billing plugin, agent-auth plugin). A legacy HS256 JWT bridge exists for the FastAPI side — the server accepts both and emits a principal to the orchestrator. Passkeys land via @better-auth/passkey.

    participant B as Browser
    participant NA as Next.js /auth/[path]
    participant BA as better-auth (EdDSA/JWKS)
    participant NP as Neon (ba_* tables)
    participant API as Sardis API (FastAPI)
    participant JWT as Legacy JWT (HS256)

    B->>NA: POST /auth/sign-in (email+passkey)
    NA->>BA: authenticate
    BA->>NP: read / write session
    BA-->>NA: session cookie (EdDSA)
    NA-->>B: 302 /(dashboard)

    B->>API: request with session cookie
    API->>BA: verify via JWKS
    API->>JWT: (fallback) verify HS256 for SDK clients
    API-->>B: response (principal attached)
