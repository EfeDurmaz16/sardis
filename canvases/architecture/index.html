Canvas 01 · System

Architecture Overview

Sardis is a seven-layer stack. An AI agent enters at the top; real money moves at the bottom. Every request traverses the same spine: identity, mandate, policy, orchestration, execution, settlement, audit. This canvas shows that spine and why each layer exists.

TL;DR AI agents can reason, but they cannot be trusted with money. Sardis inserts a verifiable control plane between an agent's decision and a bank rail — non-custodial MPC wallets below, natural-language spending policies above, an append-only audit trail on the side. Live on Base and Tempo mainnet.
Principles

Five commitments the architecture is shaped around.

These aren't slogans; each one forces a specific structural choice. Change a principle and the stack reshapes around it.

Non-custodial first
Private keys never live in Sardis memory. All signing goes through an MPC provider (Turnkey default, Fireblocks/Circle/Lit pluggable). Why it matters: an agent compromised through prompt injection cannot exfiltrate funds because there is no key to steal. A platform breach limits blast radius to metadata, not movement.
Policy before execution
Every transaction passes through PaymentOrchestrator → PreExecutionPipeline (policy + KYC/AML + mandate + rate limits + anomaly). Why it matters: there is no "private fast path" — wallets, mandates, ap2 routers all funnel through the same gate, so one hardening improves safety everywhere.
Fail-closed by default
Compliance failure, policy failure, unreachable KYA/FIDES service — all default to deny. Why it matters: agents run unattended; "when in doubt, allow" is how you wake up to a regulator letter. The override flag (SARDIS_AGIT_FAIL_OPEN=true) exists for local dev only.
Audit everything, anchor it on-chain
Every decision writes a ledger entry; batches are Merkle-rooted and anchored via SardisLedgerAnchor.sol. Why it matters: agent disputes resolve to cryptographic evidence, not logs that could have been edited.
Provider abstraction
Every external service — MPC, KYC, AML, card issuer, fiat rail, PSP — sits behind an interface in packages/sardis-core. Why it matters: Didit replaced Persona without touching business logic; Lithic can replace Stripe Issuing when funding requirements allow. Swap cost stays low, lock-in stays zero.
Open-core moat
Framework SDKs, protocol verifiers, reference contracts — open source. Policy engine, trust graph, enterprise compliance — proprietary. Why it matters: distribution goes viral through SDKs; revenue defends through the parts regulated entities actually pay for.
The Spine

Seven layers — top to bottom.

Read top-to-bottom for a request's journey; bottom-to-top for a money movement's journey. Canvas 07 · Payment Flow walks one request through every box.

Layer 1 · Agent Surface
AI framework SDKs + MCP + CLI
Where the agent code actually lives. Thin adapters that expose Sardis primitives (pay, policy check, wallet balance) as native tools in each framework. Zero crypto concepts leak up.
sardis-langchain sardis-crewai sardis-openai-agents sardis-agent-sdk sardis-adk sardis-agentkit sardis-mcp-server sardis-cli + 10 more
↓ HTTPS / JSON
Layer 2 · API Gateway
FastAPI · 123 routers · auth + rate limit + CSP
Composition root: packages/server-api/src/sardis/main.py. Every router is grouped by domain (auth, wallets, payments, policy, checkout, compliance, ops). CSP/HSTS/CORS enforced at middleware; principal extraction in require_principal; rate limits Redis-backed.
auth / api_keys / me wallets / agents / groups payments / pay / ap2 / mandates checkout / merchant_checkout compliance / kyc_onboarding admin / emergency / kill_switch
↓ PaymentOrchestrator.execute()
Layer 3 · Pre-Execution Pipeline
Policy + Compliance + Mandate + Limits + Anomaly
The single gate. Implements AGIT (Agent Identity and Trust), KYA (Know Your Agent), FIDES (trust graph), and the Spending Policy engine. Fail-closed; every deny is audited with reason code.
spending_policy.py agit_policy_engine.py anomaly_detection.py approval_service.py rate_limiter erc8183_policy_guard.py
↓ approved intent
Layer 4 · Protocol Verification
AP2 · TAP · x402 · A2A mandate chains
If the request arrived via a standard protocol (Google/Visa/Mastercard AP2, Visa TAP, x402 402-Payment-Required, Google A2A), the full mandate chain is verified — Ed25519/ECDSA-P256 sigs, replay cache, TTL — before the orchestrator will even schedule a transaction.
sardis-protocol MandateVerifier ReplayCache (Postgres + Redis) MandateArchive
↓ signed mandate
Layer 5 · Orchestration + Wallet
ChainExecutor · WalletManager · MPC signing
Chooses the rail (Base, Tempo, Ethereum, Polygon, Arbitrum, Optimism), selects the token (USDC, EURC, USDT, PYUSD), requests an MPC signature from Turnkey (or Fireblocks/Circle), and submits to the appropriate RPC. Dedicated executors for Tempo (tempo/executor.py) and EVM chains.
sardis-chain/executor.py sardis-wallet/manager.py tempo_keychain.py Circle Paymaster (USDC gas) Pimlico (backup)
↓ broadcast tx
Layer 6 · Settlement
CCTP v2 · Fiat rails · Stripe Connect
Cross-chain USDC via Circle CCTP v2 (unified addresses). Fiat off-ramp via Stripe Connect (US), Striga (EU vIBANs + SEPA), Bridge (multi-rail). Settlement is explicit and reconciled — not "best effort."
CCTP v2 burn/mint merchant_payout striga_vibans bridge (ACH/SEPA/PIX) reconciliation_queue
↓ state hashes
Layer 7 · Audit + Ledger
Append-only ledger · Merkle anchor · Evidence export
Every decision — approved, denied, errored — writes a ledger row. Batches are Merkle-rooted and anchored on-chain via SardisLedgerAnchor.sol. Operators and regulators get cryptographic evidence, not "trust our logs."
sardis-ledger SardisLedgerAnchor.sol policy_decisions evidence_export data_export

Planes

Data plane vs. control plane.

A common source of confusion: Sardis has two planes. The data plane moves money. The control plane moves opinions about money — policies, approvals, trust graph updates, anomaly thresholds. They share a database but not a code path.

Data plane — "do the thing"
API gateway → orchestrator → wallet → chain. Synchronous, latency-sensitive (agents are watching a response), fail-closed. Owned by sardis-api, sardis-chain, sardis-wallet.
Latency budget
~2s p99 end-to-end (excluding chain confirmation)
Blast radius on bug
money moves wrong → fail-closed design contains
Owner test suite
tests/integration + on-chain replay
Control plane — "decide the policy"
Policy manager + approvals + kill switch + trust registry + webhooks + live events. Async, eventually consistent, human-in-the-loop friendly. The hosted dashboard is private product code; the public repo keeps the API, policy modules, event streaming routers, and integration contracts.
Latency budget
~5s to UI from event bus
Blast radius on bug
bad policy → caught at next execution (fail-closed)
Owner test suite
Playwright (dashboard) + policy unit tests
