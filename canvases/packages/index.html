Canvas 03 · System

Public Packages

A unit ships as a separate package when its boundary is real — a provider that can be swapped, a framework adapter with its own release cadence, or a protocol surface that external auditors will pin to a version. This canvas maps the OSS protocol, SDK, and integration packages that remain public after the hosted product UI moved private.

TL;DR Core platform (12 packages) carries every line of money-moving logic. Framework integrations (19) are thin adapters that mostly forward to the Python SDK. Payment providers (6) sit behind uniform interfaces in sardis-core. SDKs + CLIs are the public surface. Hosted dashboard and checkout UI packages are intentionally not part of this public package catalog.
Dependency shape

How the core packages depend on each other.

Arrows point from consumer to provider. sardis-core is the hub — every other package either imports from it or registers a provider into it. The API package composes everything.

Group 1 · Core platform

The 12 packages that do the actual work.

If you removed any of these, the product would stop functioning. These carry domain logic, provider abstractions, and runtime enforcement. All Python, all workspace-local.

sardis-coreconfig.py, spending_policy.py, orchestrator.py, agit_policy_engine.py, anomaly_detection.pyDomain models, config, policy engine, orchestrator. Every other package imports from here. Split out so downstream can upgrade policy logic without touching transport.
sardis-apipackages/reference-api/sardis_server/routes/<domain>/*.py, packages/reference-api/sardis_server/routing/*.py, packages/reference-api/sardis_server/main.py, packages/reference-api/alembic/FastAPI composition root, domain route modules, route registrars, and Alembic migrations. The only package that binds to a port. Separated so contributors can iterate routes without rebuilding policy.
sardis-chainexecutor.py (v2), safe_account.py, smart_router.py, cctp_forwarding.py, tempo/executor.pyMulti-chain transaction execution. Dedicated Tempo executor lives here. Isolated from wallet so chain-specific RPC quirks don't leak into signing.
sardis-walletmanager.py, tempo_keychain.py, tempo_accounts.py, multisig.py, circle_client.py, session_manager.py, key_rotation.pyMPC signing abstraction (Turnkey / Fireblocks / Circle / Lit). Separated because swapping custody providers must be a zero-day change.
sardis-protocolap2.py, tap.py, x402_settlement.py, erc8122.py, erc8126.py, replay_cache_redis.py, nonce_registry.pyAP2, TAP, x402, ERC-8122/8126/8033 mandate-chain verifiers + replay protection. Pinned independently so external auditors can version-lock the verifier.
