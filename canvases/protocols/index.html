Multi-Protocol Gateway

Agent payments aren't converging on one standard yet. Google, Visa, Mastercard, Coinbase, Circle, and Stripe each have their own protocol. Sardis speaks all of them behind a single orchestrator so a merchant, SDK, or agent doesn't pick a team. One /pay front door; the right verifier underneath.

TL;DR Eight protocols wire into the same PaymentOrchestrator: AP2, TAP, x402, A2A, ACP, MPP (Stripe/Tempo), ERC-8004, ERC-8183. Every one of them terminates at the same fail-closed gate, runs on the same ledger, and emits the same webhook envelope.
The gateway

One entry, many verifiers.

Callers signal their protocol via route (/v1/ap2/..., /v1/x402/..., /v1/a2a/...) or via headers. Each protocol has a dedicated verifier in packages/sardis-protocol/; verified intents feed into the same orchestrator that wallet-native calls use.

Eight protocols

Same structure for each.

What it is · who runs it · when to reach for it · where Sardis implements it · the signature shape.

AP2 — Agent Payment Protocol

consortium
Mandate chain protocol: Intent → Cart → Payment. Each mandate is signed by the previous actor (user authorizes agent, agent authorizes merchant, merchant authorizes rail). The verifier walks the whole chain, checking Ed25519 / ECDSA-P256 signatures, TTLs, and replay cache before the transaction is even scheduled.
Who
Google · PayPal · Mastercard · Visa consortium
Use when
B2B agent commerce · multi-party flows with human approval upstream
Router
routers/ap2.py → /v1/ap2/payments/execute
Verifier
sardis_protocol/verifier.py · mandate_cache.py
POST /v1/ap2/payments/execute
{
  "intent_mandate":  { "user_did", "agent_did", "scope", "sig" },
  "cart_mandate":    { "cart", "merchant_did", "sig" },
  "payment_mandate": { "amount", "rail", "sig" }
}

TAP — Trust Anchor Protocol

visa
Visa's agent-identity attestation protocol. mTLS + TRID (Transaction Risk Indicator) for card-rail transactions. The identity piece lives in packages/sardis-protocol/src/sardis_protocol/tap.py; webhooks come in via routers/visa_tap_webhooks.py. Pairs with AP2 when Visa is the rail.
Who
Visa
Use when
Visa card rail is part of the flow (card-present or card-not-present agent purchase)
Router
routers/visa_tap_webhooks.py
Module
sardis_protocol/tap.py · tap_keys.py
agent_id + tap_key (Ed25519)
  → challenge-response attestation
  → TRID score forwarded to Visa auth
  → rail approves / declines

x402 — HTTP 402 auto-pay

coinbase · circle
Server responds 402 Payment Required with a payment challenge; client auto-pays in USDC and retries. Revives HTTP's long-dormant 402 as the canonical pay-per-call path for AI APIs. Sardis implements the verifier, settler, and challenge generator, with ERC-3009 transfer support.
Who
Coinbase + Circle Gateway facilitators
Use when
API-to-API pay-per-call · AI inference · tool marketplaces
Router
routers/x402.py → /verify · /settle · /challenges · /dry-run
Modules
x402.py · x402_erc3009.py · x402_settlement.py · x402_upto.py
GET /expensive-endpoint
← 402 Payment Required
  { "accepts": [{ "scheme": "exact", "asset": "USDC", "amount": "0.05" }] }
→ sign ERC-3009 transfer → POST /verify
→ receipt attached → retry original call

A2A — Google Agent-to-Agent

google
Google's open protocol for agents from different platforms to discover each other, trust each other, and transact. Sardis provides /pay, message envelopes, and a full trust-table API (peer discovery, relation CRUD, security-policy queries, audit).
Who
Google (open)
Use when
Agents from different platforms transacting
Router
routers/a2a.py + routers/a2a_payments.py
Key endpoints
/pay · /messages · /trust/table · /trust/check · /trust/peers · /trust/relations · /trust/security-policy · /trust/audit/recent
POST /v1/a2a/pay
{ "from_agent_did", "to_agent_did", "amount", "currency", "reason" }
→ trust_check(from_agent, to_agent)
→ mandate issued if trust ≥ threshold
→ settlement via Sardis orchestrator

ACP — Agent Commerce Protocol

emerging
Emerging standard for agent-mediated commerce — catalog, offer, order, settle. Sardis ships the router; the protocol is deliberately thin so it can ride Sardis' own orchestrator + ledger machinery rather than reinventing them.
Who
Emerging · multi-vendor spec
Use when
Catalog-first flows where discovery precedes payment
Router
routers/acp.py
Verbs
offer · order · settle · attestation · payment-proof · dispute
POST /v1/acp/offers   — merchant posts
POST /v1/acp/orders   — agent orders
POST /v1/acp/settle   — orchestrator settles
POST /v1/acp/attest   — evidence anchored

MPP — Merchant Payment Protocol

stripe · tempo
Stripe's protocol layered on Tempo (100K+ TPS EVM L1). Sessions, execute, close, policy evaluate/simulate, card issuance. Sardis is live on Tempo mainnet (chain_id 4217) via the dedicated executor in packages/sardis-chain/src/sardis_chain/tempo/ and plugs MPP sessions into Sardis policy at mpp_gate.
Who
Stripe + Tempo
Use when
High-throughput merchant flows · card issuance at scale
Router
routers/mpp.py
Endpoints
/sessions · /sessions/{id}/execute · /sessions/{id}/close · /evaluate · /simulate · /cards/issue
POST /v1/mpp/sessions                → create
POST /v1/mpp/sessions/{id}/execute   → pay
POST /v1/mpp/evaluate                → policy decision only
POST /v1/mpp/cards/issue             → issue a card

ERC-8004 — Agent attestation

erc
On-chain attestation standard for agent identity. Provides verifiable claims about agent code hash, operator, and capabilities. Sardis implements the reader/writer at packages/sardis-core/src/.../erc8004.py; it feeds into KYA and FIDES signals consumed at the gate.
Who
ERC standards track
Use when
Regulated deployments needing on-chain attestation of agent code + operator
Module
sardis_v2_core/erc8004.py
Pairs with
KYA lifecycle · FIDES DIDs · AGIT repo hash
agent register → on-chain attestation
  { agent_did, code_hash, operator_addr, sig }
reader verifies → feeds KYA + FIDES signals

ERC-8183 — On-chain policy + reputation

erc
On-chain policy module + reputation registry + job market for agents. Three modules: erc8183_policy_guard.py (enforces policy at chain edge), erc8183_reputation.py (tracks post-trade reputation), erc8183_job.py (job lifecycle). Launched with conservative caps (1% fee, USDC-only, trust hooks disabled) — audit-driven relaxation path.
Who
ERC standards track · Sardis-authored implementation
Use when
On-chain agent jobs · cross-platform reputation portability
Modules
erc8183_policy_guard.py · erc8183_reputation.py · erc8183_job.py
Caps
≤1% fee · USDC-only · trust hooks disabled pre-audit
Job posted → policy_guard evaluates
  → agent accepts → work → settle
  → reputation registry updates score
Decision matrix

Shared primitives

What every verifier reuses.

The reason supporting eight protocols is tractable: they all reduce to the same three problems (signature verification, replay prevention, mandate archival). Sardis builds those once.

ModuleRole
sardis_protocol/verifier.pyEd25519 / ECDSA-P256 / secp256k1 signature verification core.
sardis_protocol/mandate_cache.pyCanonical mandate store — every protocol's signed payload lands here.
sardis_protocol/replay_cache_redis.pyRedis-backed replay cache. TTL per protocol; required in prod.
sardis_protocol/nonce_registry.pyPer-agent, per-protocol nonce tracking to defeat replay even without Redis.
sardis_protocol/reason_codes.pyCanonical deny-reason catalog shared across protocols and audit.
sardis_protocol/rate_limiter.pyProtocol-aware rate limiter (stricter for public protocols).
sardis_protocol/payment_methods.pyRail selection — USDC / ERC-3009 / Visa / card / Lightning.
sardis_protocol/fides_signing.pyFIDES DID signing adapter used by AP2 and A2A.
sardis_protocol/eas_kya.pyEAS (Ethereum Attestation Service) bridge for KYA facts.
sardis_protocol/zkpass_transgate.pyzkPass TransGate — zero-knowledge claims surfaced into protocols.
sardis_protocol/paladin_privacy.pyPaladin privacy layer for sensitive mandate fields.
sardis_protocol/kleros.pyKleros dispute adapter — protocol-level dispute resolution.
sardis_protocol/schemas.pyCross-protocol Pydantic schemas; one definition per shape.
