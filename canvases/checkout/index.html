Canvas 11 · Product

Checkout & Merchants

The public repo keeps the checkout session engine, hardening migrations, API routes, and merchant SDK contracts. Hosted browser checkout UI and embed implementation live in the private product repo until Sardis deliberately publishes that widget as a stable OSS surface.

TL;DR A merchant creates a merchant_checkout_session. A customer receives a client_secret, presents it to a checkout client, connects a wallet or uses a managed wallet path, signs a USDC transfer, and Sardis streams the outcome back. All public endpoints are client_secret-scoped so sessions cannot be enumerated and double-pay races cannot land.
Products

One session engine, two front doors.

Both products create rows in the same merchant_checkout_sessions table and emit the same webhook envelope. The difference is who holds the wallet and where the crypto knobs live.

Pay with Sardis

Crypto-native checkout — customer brings the wallet.

Drop-in checkout, similar in shape to Stripe Checkout but paying in USDC. Customer connects MetaMask / Coinbase Wallet / any WalletConnect provider, or fiat-onramps into a fresh wallet via Coinbase CDP. Settlement lands on the merchant's configured address (or connected merchant bank account for USD payout).

  • Client-secret-scoped API for browser or server checkout clients
  • External wallet confirmation and server-managed payment paths
  • Optional onramp integration for fiat-first customers
  • EIP-191 signature binds payer wallet to session
  • SSE stream for real-time status in the merchant UI
Sardis Connect

Zero-crypto merchant SDK — three lines of code.

For API providers who want to charge AI agents without ever touching a chain. @sardis/connect wraps an Express/Next.js route, serves a /.well-known/sardis.json manifest, and reports usage to Sardis. The merchant sees a USD balance; Sardis handles wallet creation, settlement, and payout.

  • packages/sardis-connect (Python) · packages/sardis-connect-js (TS)
  • PricedEndpoint / UsageRecord / ServiceManifest models
  • Protocol-agnostic: accepts: ["sardis", "x402", "mpp"]
  • Per-call, per-unit, and subscription pricing tiers
  • Payout via connected merchant bank · no wallet UX on the merchant side
Pay with Sardis · End-to-end

Seven steps from merchant intent to settled USDC.

Every step corresponds to a real endpoint in apps/api/server/routes/commerce/merchant_checkout.py. Public endpoints live under the /sessions/client/{client_secret}/... namespace; merchant endpoints require a principal.

01
Merchant creates session. POST /v1/merchant/checkout/sessions (authenticated with merchant principal) → inserts a row into merchant_checkout_sessions with amount, currency, chain, settlement target, optional idempotency_key and embed_origin. Server generates a client_secret (24 random bytes, base64).
02
Redirect or embed. Merchant passes the client_secret to a checkout client. Hosted UI is product-private; the public contract is the client-secret-scoped session API and the webhook/SSE contract.
03
Checkout client loads session. A browser, mobile, or server checkout client fetches GET /sessions/client/{client_secret}/details — returns the amount, merchant name, and the settlement_address needed for client-side USDC transfers.
04
Wallet connect. Customer connects via wagmi. Two paths converge here: external wallet (POST /connect-external, EIP-191 signature verified server-side) or CDP Onramp (GET /onramp-token, fiat → USDC in a fresh wallet). Either way, payer_wallet_address is persisted on the session.
05
Payment. Customer signs a USDC transfer to settlement_address. Frontend submits via viem; server records the tx hash via POST /confirm-external-payment. Session flips pending → confirmed.
06
Streaming update. Merchant systems and checkout clients subscribe to GET /sessions/client/{client_secret}/stream (SSE). Polling fallback at .../details exists for clients that can't hold an SSE connection.
07
Settlement + webhook. Background job reconciles the on-chain receipt, computes platform_fee_amount and net_amount, flips status to settled, and enqueues a checkout.session.settled webhook. Delivery is tracked in merchant_webhook_deliveries keyed by event_id for dedup.
State machine

Merchant checkout session lifecycle.

The session row moves through a narrow set of states. Every transition is audited; every terminal state triggers a webhook. 

stateDiagram-v2
    [*] --> created : POST /merchant/checkout/sessions
    created --> pending : client loads /s/{client_secret}
    pending --> expired : TTL elapsed
    pending --> connected : connect-external (EIP-191 verified)
    pending --> connected : onramp-token (CDP, payer wallet bound)
    connected --> confirmed : confirm-external-payment (tx hash)
    confirmed --> settled : reconciliation job · receipt verified
    settled --> paid_out : connected merchant bank transfer (if USD payout)
    pending --> failed : signature / balance / policy rejection
    confirmed --> failed : tx reverted on-chain
    settled --> [*]
    paid_out --> [*]
    expired --> [*]
    failed --> [*]
Hardening

Why public checkout is boring in a good way.

apps/api/migrations/031_checkout_hardening.sql is the migration that turned Pay with Sardis from "it works on the happy path" into something safe to point at the open internet. Polar-inspired; every column earns its keep.

ControlColumn / TableWhat it prevents
Client secret merchant_checkout_sessions.client_secret (UNIQUE) Session ID enumeration. Public endpoints use /sessions/client/{client_secret}/..., never the internal UUID.
Idempotency key merchant_checkout_sessions.idempotency_key (UNIQUE, partial) Duplicate session creation on merchant-side retry. DB-level uniqueness, not app-level.
Row lock on pay SELECT ... FOR UPDATE NOWAIT in the pay path Double-pay races when a customer clicks twice or two tabs submit simultaneously.
Embed origin merchant_checkout_sessions.embed_origin Clickjacking / hostile iframe. Hosted clients must restrict postMessage to the declared origin.
Webhook dedup merchant_webhook_deliveries.event_id (UNIQUE) Double-processing on retries. Merchants can safely use at-least-once delivery.
Fee transparency platform_fee_amount · net_amount "Why did I get less than I charged?" support tickets. Full breakdown on every session.
Reusable links merchant_checkout_links table One-off sessions for static storefronts; merchant generates a slug, Sardis creates a fresh session per visitor.
Payer binding payer_wallet_address (migration 033) Wrong-wallet payment. The onramp token endpoint validates the wallet matches the session binding.
Hosted client boundary

What remains private vs. public.

The public repo documents and tests the backend contract. The hosted React checkout client, visual design, and embeddable script are product code and should be recovered into the private product repo from git history if needed.

  • Client-secret-scoped public session operations.
  • SSE status stream and webhook event envelope.
  • Database idempotency, deduplication, and settlement state.
Private product surface

Two ways to fund a payment

External wallet · CDP Onramp.

Customers either already own crypto (external wallet path) or don't (CDP Onramp path). Both converge at the same session state: a verified payer_wallet_address with enough USDC to settle.

External wallet — EIP-191 binding
Frontend: wagmi + viem + WalletConnect v2 + Coinbase Wallet SDK. Flow: GET /connect-params returns the challenge to sign, customer signs with wallet, frontend posts signature to POST /connect-external. Server verifies the EIP-191 signature recovers the claimed address and persists it as payer_wallet_address. On pay, customer signs the actual USDC transfer; frontend reports the hash via POST /confirm-external-payment.
CDP Onramp — fiat to USDC
For customers without crypto. Sardis mints an MPC wallet for the session, calls POST api.developer.coinbase.com/onramp/v1/token with a CDP JWT, and returns a hosted pay.coinbase.com/buy/select-asset URL bound to that wallet. The /onramp-token endpoint validates the wallet address matches the session — an attacker cannot redirect funds to a different wallet by tampering with the query string.
Sardis Connect

Three lines of middleware, zero crypto on the merchant side.

Sardis Connect is the opposite philosophy to Pay with Sardis. The merchant never sees a wallet, never sees chain config, never handles stablecoin balances. They wrap their API with sardisConnect({ apiKey }), declare pricing, and get USD in their connected merchant bank account. Sardis absorbs the crypto layer entirely.

Merchant code — TypeScript
import { sardisConnect } from '@sardis/connect';
const sardis = sardisConnect({ apiKey: 'mch_live_xxx' });
app.use(sardis.middleware());
The middleware reads a priced endpoint manifest, rejects unpaid calls with 402 Payment Required (x402-compatible), and forwards paid calls to the merchant handler. Protocol-agnostic: one middleware accepts Sardis native, x402, and MPP.
Service manifest
Served at /.well-known/sardis.json — agents discover pricing via ServiceManifest:
  • PricedEndpoint(path, method, price, pricing_model)
  • pricing_model: per_call · per_unit · subscription
  • UsageRecord for metered billing (report units after the call)
  • Tiered pricing via PricingTier
Risk + merchant-side controls

Trust scoring and the merchant kill switch.

Two pieces of machinery sit next to the session engine: per-merchant trust profiles (Sardis decides how much to trust the merchant) and checkout controls (the merchant decides what shape of payments to accept).

Merchant trust — migration 045
merchant_trust_profiles stores trust_level, a rolling trust_score (default 0.3), transaction_count, total_volume, and dispute_count. MerchantTrustService adjusts approval thresholds: first-seen merchants face stricter KYC + lower velocity caps, merchants with a dispute history get flagged for manual review. Verified merchants (verified_at, verified_by) get the tightest loop.
Checkout controls router
apps/api/server/routes/commerce/checkout_controls.py exposes merchant-side knobs: amount caps, country allowlists, velocity limits, required customer-KYC tier. GET /config / PUT /config for configuration, POST /incidents / GET /incidents for the merchant's audit trail of auto-blocked or flagged sessions.
