## What you'll build

A script that creates an AP2 IntentMandate, executes a payment against it, and demonstrates the kill switch — Sardis's emergency stop that halts every in-flight and future payment on a rail with a single call. This is the pattern you'll deploy for any agent that handles real money, because mandates plus kill switches give you both proactive (policy) and reactive (panic button) controls.

## Prerequisites

- Python 3.12+
- `pip install sardis`
- Completed [Tutorial 1](/docs/tutorials/01-your-first-agent-payment)
- A sandbox API key in `SARDIS_API_KEY`

## Step 1: Understand the mandate chain

AP2 — Agent Payment Protocol, the Google/PayPal/Mastercard/Visa standard — splits a payment into three signed steps:

```
IntentMandate ───▶ CartMandate ───▶ PaymentMandate
   (User)           (Merchant)         (Signed)
```

The user signs an intent ("I authorize up to $50 of SaaS spend"), the merchant assembles a cart that fits inside that intent, and the resulting payment carries both signatures. Sardis's verifier checks the chain end to end: signature validity, chain integrity, replay nonces, and policy fit. You don't have to implement any of that — you just create the mandates and let Sardis enforce them.

Why bother with mandates if you already have policies? Because mandates are *per-task* and *user-attributable*. A policy says "this agent can spend $100/day"; a mandate says "for *this specific shopping task*, the user agreed to $50 max in the SaaS category." When something goes wrong, mandates give you a cryptographic audit trail back to a real human decision.

## Step 2: Create the intent mandate

We'll use the async client because mandate flows usually involve concurrent verification calls.

```python
import asyncio, os
from sardis import AsyncSardis

async def create_intent(client: AsyncSardis) -> str:
    intent = await client.payments.create_intent_mandate(
        agent_id="shopping-agent-1",
        max_amount_minor=50_000_000,  # $50.00 in USDC base units (6 decimals)
        allowed_categories=["saas", "ai_api"],
        expires_in_seconds=3600,
    )
    return intent["id"]
```

A few things to know:

- `max_amount_minor` is in the token's smallest unit. USDC has 6 decimals, so $50 = 50_000_000. Using minor units everywhere eliminates floating-point bugs in money math — a hard-won lesson from every payment system that tried floats first.
- `allowed_categories` is enforced server-side against a curated merchant taxonomy. An attempt to pay a gambling site under a `saas`-only intent will return `policy_blocked` with a categorical mismatch reason.
- `expires_in_seconds` prevents stale intents from being replayed weeks later. Mandates are not bearer tokens; they expire by design.

## Step 3: Execute against the mandate

Once you have an intent ID, attach it to the payment call:

```python
async def pay_against_intent(client: AsyncSardis, intent_id: str) -> dict:
    return await client.pay.execute(
        to="api.openai.com",
        amount="12.50",
        currency="USDC",
        mandate_id=intent_id,
    )
```

The presence of `mandate_id` flips Sardis into mandate-verification mode. It pulls the intent, verifies the user signature, checks that the requested merchant fits the allowed categories and the requested amount fits inside the remaining mandate balance, then settles. A single intent can back several payments until it's exhausted or expired — useful for an agent that pays in installments as it works through a task.

If verification fails, the response includes a `mandate_reason` field naming the exact predicate that rejected the call (`category_mismatch`, `amount_exceeded`, `expired`, `signature_invalid`). Log it; don't swallow it.

## Step 4: Wire up the kill switch

Mandates protect you when the policy logic works. The kill switch protects you when *the agent itself* misbehaves — prompt injection, model regression, runaway loop, compromised credentials. It halts every payment on a rail in one call:

```python
async def emergency_stop(client: AsyncSardis, reason: str) -> None:
    await client.kill_switch.activate_rail(
        rail="base",
        reason=reason,
    )
```

After this returns, every subsequent `pay.execute()` on the Base rail returns `rail_disabled` until you explicitly call `deactivate_rail`. There's no automatic timeout — re-enabling is a deliberate human action, which is exactly what you want during an incident.

Production tip: bind the kill switch to your on-call alerting system. A "spend velocity over $X/min" alert in PagerDuty that ships a button to call `activate_rail` turns a multi-hour incident into a 30-second response.

## Step 5: Check status

Before you build any retry or fallback logic, check whether a rail is live:

```python
async def is_live(client: AsyncSardis, rail: str) -> bool:
    status = await client.kill_switch.status()
    return rail not in status.get("disabled_rails", [])
```

Agents should consult this before kicking off a long-running task, not just before each payment — it's better to fail at task-plan time than after burning two minutes of LLM tokens.

## Final: full working example

```python
import asyncio, os
from sardis import AsyncSardis

async def main() -> None:
    async with AsyncSardis(
        api_key=os.environ["SARDIS_API_KEY"],
        base_url="https://sandbox.api.sardis.sh",
    ) as client:
        intent = await client.payments.create_intent_mandate(
            agent_id="shopping-agent-1",
            max_amount_minor=50_000_000,
            allowed_categories=["saas", "ai_api"],
            expires_in_seconds=3600,
        )
        print(f"Intent created: {intent['id']}")

        receipt = await client.pay.execute(
            to="api.openai.com",
            amount="12.50",
            currency="USDC",
            mandate_id=intent["id"],
        )
        print(f"Payment status: {receipt['status']}")

        if receipt["status"] == "succeeded":
            print(f"  tx={receipt['tx_hash']}")
        else:
            print(f"  reason={receipt.get('mandate_reason') or receipt}")
            await client.kill_switch.activate_rail(
                rail="base",
                reason="unexpected mandate failure — investigating",
            )
            print("Kill switch ACTIVE on base rail")

if __name__ == "__main__":
    asyncio.run(main())
```

Run it against sandbox. You'll see the intent ID, a payment receipt, and — if anything failed — a kill switch activation. Re-enable the rail from the dashboard or with `kill_switch.deactivate_rail` once you're satisfied.
