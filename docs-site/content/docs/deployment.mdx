## What This Guide Covers

This guide is for contributors and operators who want to run the open-source Sardis API themselves.

It covers:

- Local API execution
- Container builds
- Google Cloud Run deployment through GitHub Actions and Cloud Build
- Required runtime configuration
- Public validation commands

It does not cover hosted Sardis Cloud operations, production customer provisioning, managed secrets, partner onboarding, sales demos, or dashboard administration.

## Local API

Install dependencies from the repository root:

```bash
uv sync
pnpm install --frozen-lockfile
```

Start the API locally:

```bash
uv run uvicorn sardis.main:create_app --factory --host 0.0.0.0 --port 8000
```

Verify the health endpoint:

```bash
curl -sf http://localhost:8000/health
```

## Container Build

The root `Dockerfile` builds the public FastAPI backend with the required local Python packages installed into the image.

```bash
docker build -t sardis-api:local .
docker run --rm -p 8000:8000 \
  -e SARDIS_ENVIRONMENT=development \
  sardis-api:local
```

Then verify:

```bash
curl -sf http://localhost:8000/health
```

## Runtime Configuration

Minimum local development configuration:

| Variable | Required | Notes |
| --- | --- | --- |
| `SARDIS_ENVIRONMENT` | No | Use `development`, `staging`, or `production`. |
| `DATABASE_URL` | Production | Required in production. Use PostgreSQL. |
| `SARDIS_REDIS_URL` | Production | Preferred Redis URL for rate limiting, idempotency, and revocation. `REDIS_URL` and `UPSTASH_REDIS_URL` are also accepted. |
| `SARDIS_SECRET_KEY` | Production | Secret material for server-side signing/encryption paths. Store in a secret manager. |
| `JWT_SECRET_KEY` | Production | Secret used for JWT/session validation. Store in a secret manager. |
| `SARDIS_TAP_JWKS_URL` | Production | Required for TAP authentication in production. |

Provider-specific variables are optional until the corresponding rail or integration is enabled. Keep provider credentials in your deployment secret manager, not in repository files.

## Cloud Run Reference Path

The public repo ships a GitHub Actions workflow at `.github/workflows/deploy-api-cloudrun.yml` and a Cloud Build config at `cloudbuild.yaml`.

Required GitHub secrets:

| Secret | Purpose |
| --- | --- |
| `GCP_WORKLOAD_IDENTITY_PROVIDER` | Workload Identity Federation provider for GitHub Actions. |
| `GCP_SERVICE_ACCOUNT` | Service account used by the deploy workflow. |
| `GCP_PROJECT_ID` | Target Google Cloud project. |

The workflow:

1. Authenticates with Google Cloud through Workload Identity Federation.
2. Builds the root `Dockerfile` through Cloud Build.
3. Pushes `gcr.io/$GCP_PROJECT_ID/sardis-api:latest`.
4. Deploys the image to Cloud Run.
5. Verifies `/health`.

For a manual equivalent:

```bash
gcloud builds submit \
  --project="$GCP_PROJECT_ID" \
  --config=cloudbuild.yaml \
  --substitutions=_SERVICE_NAME=sardis-api

gcloud run deploy sardis-api \
  --project="$GCP_PROJECT_ID" \
  --region=us-east1 \
  --image="gcr.io/$GCP_PROJECT_ID/sardis-api:latest" \
  --platform=managed \
  --allow-unauthenticated \
  --port=8000
```

Set production secrets through Cloud Run, Secret Manager, Terraform, Pulumi, or your internal deployment system. Do not add deployment credentials, customer keys, or staging bootstrap scripts to this repository.

## Public Validation

Before opening a deployment-related pull request, run:

```bash
python3 scripts/oss_surface_check.py
python3 scripts/package_maturity_check.py
pnpm --filter @sardis/app-landing typecheck
pnpm --filter canvas-site build
pnpm run verify
```

For full release readiness:

```bash
pnpm run check:release-readiness
```

## API Keys

For the hosted API, use the public API-key flow documented in [Get Your API Key](/docs/get-api-key).

For self-hosted deployments, create operator/admin keys through your own deployment bootstrap path and keep that bootstrap code private if it embeds customer, staging, or organization-specific assumptions. Public contributions should focus on stable API behavior, schema, SDKs, tests, and protocol compatibility.
